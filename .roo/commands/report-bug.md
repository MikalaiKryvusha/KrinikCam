# /report-bug — завести баг-документ в `bugs/` (агент сам ведёт беклог багов)

Всякий раз, когда агент пишет код, тестирует его и **сталкивается с багом** (краш, неверное поведение,
артефакт, регресс, дефект библиотеки), он заводит ОТДЕЛЬНЫЙ md-документ в `bugs/` по тем же правилам,
что и уже существующие баг-доки. Так агент сам для себя копит беклог багов — ничего не теряется, и
к багу можно вернуться (или отдать его в `/bug-research`).

> Правило проекта (AGENT_GUIDE): «по всем, даже мелким багам в процессе работы — рефлексируй и собирай
> знания в отдельных md в `bugs/`». Этот навык автоматизирует именно это.

## Когда вызывать

- Агент в ходе разработки/теста увидел дефект, который НЕ чинит прямо сейчас (или чинит, но хочет
  зафиксировать форензику/постмортем).
- Нужно отложить баг (взять другую задачу), не потеряв его.
- Криник просит оформить баг.
- НЕ для «затыка от непонимания задачи» (это про `PHILOSOPHY.md`) и не вместо фикса тривиальной опечатки.

## Что сделать

1. **Определи следующий номер.** `ls bugs/` → максимальный `NN` (двузначный) + 1. Имя файла:
   `bugs/NN_<короткое_имя_на_английском>.md` (snake_case, как у соседей; БЕЗ тега `DONE` — баг открыт).

2. **Собери факты ДО написания** (не выдумывай — баг-док ценен фактами):
   - Симптом: что именно наблюдается (и как это отличается от ожидаемого).
   - Репро: детерминированные шаги. По возможности — через харнесс `node tools/ui.mjs cmd …`
     (virtual-camera/compositor/go-live/stop/…), чтобы баг воспроизводился без Криника.
   - Форензика: `adb logcat` / `adb logcat -b crash` / краш-файл приложения / `ffprobe` / кадр —
     приложи ключевые строки (стек, abort message, EGL/GL ошибки, размеры).
   - Версия билда (`version.json` / последний коммит), устройство, режим (компоновщик вкл/выкл и т.п.).

3. **Напиши документ по канону** (структура как у `bugs/13..20`):
   ```
   # Bug NN — <краткое описание одной строкой>

   **Статус:** 🔴 ОТКРЫТ   (или 🟡 частично / 🔬 только исследование / 🔧 фикс есть, ждёт проверки)
   **Версия:** <build>   ·   **Когда/контекст:** <дата, в ходе какой задачи нашли>

   ## Симптом
   <что наблюдается>

   ## Репро (детерминированно)
   <шаги; харнесс-команды, если есть>

   ## Форензика
   <ключевые логи/краш/ffprobe/кадр>

   ## Root cause / Гипотезы
   <причина, если ясна; иначе ранжированные гипотезы — НЕ латать вслепую>

   ## План фикса (или фикс, если сделан)
   <шаги; связь с архитектурой/другими багами>

   ## Связь
   <ссылки на related bugs/ideas/interviews>
   ```
   Руководствуйся `BUG_FIXING_FRAMEWORK.md`. Если баг — в чужой библиотеке, заведи им тикет через `gh`
   (см. AGENT_GUIDE) и сошлись на него в доке.

4. **Зафиксируй в беклоге/процессе:**
   - Если баг важен/блокирует — короткой строкой в `STATUS.md`.
   - Закоммить (в автолупах — по общей дисциплине; иначе можно батчем): `node tools/commit.mjs "docs(bugNN): …"`.

5. **Жизненный цикл:** пока баг открыт — файл БЕЗ `DONE`. Когда баг ПОДТВЕРЖДЁННО закрыт (исправлен и
   проверен) — по правилу Idea 15 переименуй `git mv bugs/NN_x.md bugs/NN_DONE_x.md` и допиши внутрь
   секцию `## ✅ СТАТУС: DONE (дата)` (что сделано / как проверено). Ревизия беклога — навык `/check-backlog`.

## Заметки
- Лучше завести баг и оставить открытым, чем потерять. Точность фактов важнее красоты текста.
- Если баг не поддаётся (≥3 неудачные попытки фикса вслепую) — переходи в `/bug-research` (исследование
  без кода) по этому же документу.
