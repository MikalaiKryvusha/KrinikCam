# /check-backlog — ревизия беклога (bugs/ и plans/) и простановка тегов DONE

Со временем в `bugs/` и `plans/` копятся файлы, которые УЖЕ сделаны, но не помечены `DONE`, —
из-за этого по списку файлов нельзя понять, что осталось. Этот навык наводит порядок: собирает
актуальный список открытых задач И проставляет `DONE` тем, что по факту закрыто.

Опирается на конвенцию тега `DONE` в имени файла (Idea 15, см. `AGENT_GUIDE.md` → «Бэклог: bugs/ и
plans/ideas/ — тег DONE в имени файла»).

## Что сделать

1. **Собрать все файлы беклога:**
   - `ls bugs/` — все баг-доки.
   - `ls plans/` и `ls plans/ideas/` — идеи/фичи и рабочие журналы/планы.
   - ⚠️ В `plans/` есть РЕФЕРЕНСНЫЕ доки (не задачи): `master_plan.md`, `project_map.md`, `goal.md`,
     `context.md`, `automation.md`, `github.md` и т.п. — их НЕ трогаем тегом DONE, это живые
     справочники, а не закрываемые задачи. Тегаем только конкретные баги/идеи/фичи/задачи.

2. **Разделить на два списка по имени файла:**
   - **Открытые (нет `DONE` в имени)** — это кандидаты в беклог «к реализации». Берём на карандаш.
   - **С тегом `DONE`** — уже закрытые, пропускаем.

3. **Для КАЖДОГО открытого файла — определить реальный статус** (не гадать!):
   - Прочитать сам документ: есть ли внутри статус ✅/ЗАКРЫТ/РЕАЛИЗОВАНО/ИСПРАВЛЕНО/ДОКАЗАНО.
   - Свериться с `STATUS.md` и при необходимости с кодом/`git log` — сделано ли это по факту.
   - **Если по факту СДЕЛАНО и подтверждено** → выполнить простановку DONE (шаг 4).
   - **Если НЕ сделано / частично / только исследование (🔧/🟡/🔬)** → оставить открытым,
     внести в актуальный беклог-список (с кратким «что осталось»).
   - **Если сомнительно/неясно** → НЕ тегать, оставить открытым и отметить как «требует проверки».

4. **Простановка DONE (только для подтверждённо закрытых):**
   - Переименовать файл, вставив `DONE` после номера, сохраняя историю:
     `git mv bugs/13_usb_detach_crash.md bugs/13_DONE_usb_detach_crash.md`
     (номер НЕ менять; формат `<NN>_DONE_<name>.md`).
   - **Внутрь документа дописать секцию статуса** — что именно было сделано, например:
     ```
     ## ✅ СТАТУС: DONE (<дата>)
     Что сделано: <короткое резюме фикса/реализации>.
     Как проверено: <билд/харнес/ffprobe/коммит-хэш/итерация цикла>.
     ```
   - Закоммитить (в автолупе — по общей дисциплине коммитов; иначе можно батчем в конце ревизии).

5. **Выдать итог:**
   - Краткий список ОТКРЫТЫХ задач (беклог к реализации) с приоритетом: довести начатое > баги >
     новые идеи.
   - Перечень файлов, которым проставили `DONE` в этой ревизии.
   - Что осталось «требует проверки» (неясный статус).

## Заметки
- Это аккуратная ревизия: **тегаем DONE только то, что РЕАЛЬНО закрыто и проверено** — лучше
  оставить открытым, чем ошибочно закрыть.
- Референсные доки в `plans/` (планы/карты/справочники) тегом DONE не помечаем.
- В автолупах (nightloop/dayloop) и при `/refresh-context` вызывай этот навык РАЗ В НЕСКОЛЬКО
  ИТЕРАЦИЙ — чтобы беклог оставался чистым, а закрытое было помечено. Не каждую итерацию.
- Если в ходе ревизии всплыли вопросы уровня Криника (UX/бренд/архитектура) — оформи в `interviews/`
  + пометь в `STATUS.md` «❓ на ревью Криника», не решай вслепую.
