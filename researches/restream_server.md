# Ресёрч: Self-Hosted ReStream Server для KrinikCam (idea 38)

> Скелет-ресёрч ночного цикла 2026-07-18 (веб-источники внизу; код не трогался).
> Идея Криника: KrinikCam льёт ОДИН поток на свой сервер (Docker на ПК с широким каналом),
> сервер размножает на N платформ → мобильный канал и батарея щадятся («respectful mobile usage»),
> всё в экосистеме KrinikCam без платных сервисов.

## 1. Почему это правильная архитектура (подтверждение тезисов идеи)

- **Один кодек/один поток с телефона** — уже наш путь (MultiStream: один энкодер). Идея 38 убирает
  последний множитель: N×битрейт аплинка. С сервером аплинк = 1×битрейт независимо от числа платформ.
- **Облачный бенчмарк** (Restream.io, июль 2026): Free — 2 канала с водяным знаком; Standard $16/мес —
  3 канала; Professional $39/мес — 5 каналов 1080p; Business $199/мес — 8 каналов + SRT. Свой сервер
  на уже имеющемся ПК = 0 ₽/мес и без ограничений каналов — сильная точка позиционирования KrinikCam.
- **Наш RTMP-полигон уже MediaMTX** (`tools/rtmp-server.mjs`) — часть стека знакома и проверена боем
  (приёмки plans/09, bug 36, idea 37 шли через него).

## 2. Кандидаты серверного ядра

| Вариант | Суть | Плюсы | Минусы |
|---|---|---|---|
| **MediaMTX** (bluenviron) | Универсальный медиасервер RTMP/SRT/RTSP/WebRTC, один бинарь/Docker | Уже наш полигон; HTTP API v3 (paths add/list/delete); hot-reload конфига; SRT на будущее | Размножение потока — НЕ нативное: паттерн `runOnReady` + N параллельных ffmpeg (`-c copy`, CPU почти не ест) |
| **datarhei Restreamer** | Готовый self-hosted рестример С web-UI (Docker), внутри FFmpeg+ядро | Самый близкий к «готовому продукту»: UI, публикация на YouTube/Twitch/…; нулевая разработка сервера | Чужой UI/экосистема — «мозг» настройки у сервера, а не у KrinikCam; интеграция «телефон настраивает сервер» потребует его API/скриптинга |
| **nginx-rtmp** | Классика: `push rtmp://…` директивы на приложение | Простой, вечный, минимум ресурсов | Конфиг статический (перезагрузка на изменение), API нет — настройка с телефона = свой обвяз |
| **SRS** | Мощный медиасервер с HTTP API | Форвардинг из коробки, API | Тяжелее в освоении; для MVP избыточен |

**Рекомендация (моя, технический дефолт): MediaMTX + тонкий «оркестратор»** — мы уже на нём, у него
честный API; размножение = `runOnReady: ffmpeg -c copy → N адресов` (CPU-дёшево: без перекодирования).
datarhei Restreamer — запасной путь «взять готовое», если захотим web-UI на сервере.

## 3. Целевая схема (MVP)

```
KrinikCam (телефон, мобильная сеть)
   │  ОДИН RTMP/SRT поток (битрейт профиля, адаптив idea 37 работает как есть)
   ▼
KrinikCam Server (Docker на ПК: MediaMTX + оркестратор)
   │  ffmpeg -c copy ×N (без перекодирования → CPU ~нулевой)
   ├──► rtmp://a.rtmp.youtube.com/live2/KEY1
   ├──► rtmp://live.twitch.tv/app/KEY2
   └──► … платформа N
```

- **Настройка с телефона**: KrinikCam по HTTP передаёт серверу список выходов (URL+ключ из наших же
  профилей) → оркестратор пишет path-конфиг MediaMTX (API `POST /v3/config/paths/add/{path}` или
  генерация mediamtx.yml + hot-reload). ⚠️ Нюанс из issues: мутация конфига пути через API рвёт
  живого паблишера — поэтому выходы настраиваем ДО старта эфира (наш флоу это позволяет: настройка →
  Go Live).
- **В приложении**: режим мультистрима «Через мой сервер» — профили платформ помечаются
  «серверные», а сам эфир идёт ОДНИМ выходом на адрес сервера. Ложится на существующую модель:
  профиль-платформа CUSTOM «KrinikCam Server» + флаг режима; изменений энкодер-пайплайна НЕ нужно.
- **Безопасность**: stream-ключи платформ уезжают НА СЕРВЕР (наш же ПК Криника) — по духу тот же
  трастовый контур; канал телефон→сервер защитить токеном пути (MediaMTX auth) и, позже, SRT с
  passphrase. Редакция ключей в логах сервера — тем же правилом redactRtmp.

## 4. Фазы внедрения (набросок)

1. **Ф0 — прототип руками (полдня):** docker-compose с MediaMTX + статический mediamtx.yml на 2
   выхода (полигон-паттерн уже наш); KrinikCam льёт один поток; проверить каналы/стабильность.
2. **Ф1 — оркестратор (1-2 дня):** маленький сервис (Node, как наши tools) рядом с MediaMTX:
   REST «установи выходы [{url,key}…]» → конфиг путей; токен-авторизация; docker-compose из одного
   `docker compose up`. Репо-структура: `server/` в этом же репозитории (моно-экосистема KrinikCam).
3. **Ф2 — интеграция в приложение:** экран «Мой сервер» (адрес+токен), режим мультистрима
   «сервер/самостоятельно», отправка выходов перед Go Live, отображение health (сервер может
   репортить состояние push-ей через MediaMTX API).
4. **Ф3 — качество жизни:** SRT-инжест (устойчивость на мобильной сети), запись на сервере,
   рестарт упавших push-ей, метрики.

## 5. Вопросы-развилки Кринику (владельческие)

1. **Ядро:** MediaMTX+свой оркестратор (наш стек, наш UX) или datarhei Restreamer (готовый web-UI,
   но чужой «мозг»)? Рекомендую первое — экосистема KrinikCam.
2. **Где живёт правда о платформах в серверном режиме:** телефон каждый раз передаёт выходы (сервер
   без состояния — проще и безопаснее) или сервер хранит свой список (телефон только жмёт Go Live)?
   Рекомендую: телефон передаёт (одна правда — профили KrinikCam).
3. **server/ в этом репо или отдельный репозиторий?** Рекомендую: `server/` здесь (моно-экосистема,
   один релизный цикл).
4. **Протокол телефон→сервер в MVP:** RTMP (есть сейчас) или сразу SRT (лучше на мобильной сети, но
   +работа)? Рекомендую: RTMP в MVP, SRT в Ф3.
