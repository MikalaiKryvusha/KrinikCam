# Idea 37 — Адаптивный битрейт + живая статистика здоровья эфира

> Статус: **✅ ОДОБРЕНА Криником 2026-07-12 (interview_009 Q5=A)** — реализация ПОСЛЕ стабилизации
> мультистрима (plans/09). Предложена агентом 2026-07-12 (из ревизии кода).
> Согласуется с GOAL «мобильный OBS» и болью Криника «Piko+ лагает» (см. также plans/08 —
> UVC-контролы решают лаг источника; эта идея решает лаг КАНАЛА).

## Суть

1. **Adaptive bitrate**: сейчас битрейт жёстко из профиля — на слабом аплинке (мобильная сеть,
   дальний Wi-Fi) RTMP-очередь пухнет, зритель видит фризы. RootEncoder уже умеет всё нужное:
   `onNewBitrate` + `streamClient.hasCongestion()` + `setVideoBitrateOnFly()` — классический
   BitrateAdapter-паттерн: при затыке канала плавно снижать битрейт (до пола ~1 Мбит/с), при
   свободном канале — плавно восстанавливать к профильному.
2. **Живая статистика эфира** в UI: сейчас `Live.durationMs`/`droppedFrames` — мёртвые поля, а
   `bitrateKbps` при мультистриме мигает между выходами. Сделать честно: таймер эфира,
   суммарный/пер-выходной битрейт, dropped frames, индикатор congestion (зелёный/жёлтый/красный)
   — маленькая строка health в углу + детали в панели платформ (стыкуется с per-output статусами
   plans/09 S2).

## Ценность для стримера

Стрим перестаёт фризить на плохой сети (деградирует качеством, а не плавностью), и Криник ВИДИТ
здоровье эфира (как в OBS: битрейт/дропы), а не только надпись LIVE.

## Границы (KISS)

Никаких графиков/гистограмм в MVP — одна строка цифр + цветной индикатор. Пороговые значения —
константы с понятными именами. Реализация ПОСЛЕ стабилизации мультистрима (plans/09), т.к. статистика
опирается на per-output state.

- Где строка health: у FAB / верхний угол / только в панели платформ? (рекомендую: верхний левый

## ✅ СТАТУС: РЕАЛИЗОВАНА (2026-07-18, ночной цикл, build 0.7(20)) — DONE

Реализовано ровно по сути идеи (дефолты из рекомендаций, Q5=A):
- **Телеметрия**: секундный тикер эфира (самозавершается по концу стрима) — durationMs тикает,
  битрейт агрегата = СУММА живых выходов, поллинг `hasCongestion()` по каждому живому выходу
  (`OutputStatus.congested`). Health-строка = прокачанный LiveBadge: «ЭФИР 12:34 · 4,2 Mbps» +
  точка здоровья (зелёная/жёлтая-затык/красная-деградация выхода) с белой обводкой.
- **Adaptive bitrate**: затык любого живого выхода → −20% (пол 1 Мбит/с), чистый канал → +10% от
  цели до профильного (`setVideoBitrateOnFly`, энкодер один на все выходы). Вкл, если ВСЕ профили
  эфира просят (тумблер в редакторе профиля, дефолт ВКЛ).
- **Миграция Room 1→2** (`adaptiveBitrate INTEGER NOT NULL DEFAULT 1`) — ПЕРВАЯ настоящая по
  строгой дисциплине plans/12: профиль, созданный на v1, пережил апдейт живьём; schemas/2.json в git.
- **Харнес**: CMD `simulate-congestion on|off` (+ ui.mjs usage, контракт-шаг smoke).

**Приёмка наблюдением (полигон, 2 выхода):** badge «ЭФИР 0:30 · 4,7 Mbps» тикает; симуляция затыка →
лог 4000→3200→2560→2048→1638→1310 kbps (шаги −20%/2с); off → 1848→2248→…→3448 (+10% к цели);
чистый стоп тикера; smoke PASS. UM §7 дополнен (EN/RU).
