# Bug 07 — ui.mjs «не видит экран»: stale-дамп при анимации (uiautomator idle)

> Статус: **✅ ЗАКРЫТ** (2026-06-28). Инструмент: `tools/ui.mjs`.
> Нашёл Криник при тестировании Settings: «экран приложения открыт, я его вижу, а инструмент — нет».

---

## Симптом

`ui.mjs dump`/`tap` возвращали элементы СТАРОГО экрана (диалог, который уже закрыт), а не текущего.
`tap "menu"` не находил FAB, координаты иногда были из другой ориентации (landscape, хотя экран
портретный). Выглядело как «инструмент ненадёжен / не видит экран».

## Root cause (две связанные причины)

1. **`uiautomator dump` молча падает на неидл-экранах.** Если на экране есть непрерывная анимация
   (у нас — бесконечный пульс логотипа в `StandbyPlaceholder`, Compose `rememberInfiniteTransition`),
   uiautomator никогда не достигает «idle» и пишет в **stderr** `ERROR: could not get idle state.`,
   возвращая ненулевой/пустой результат и **НЕ создавая** файл дампа.
2. **Инструмент тянул устаревший файл.** `dumpUi()` делал `uiautomator dump` → `pull /sdcard/ui_dump.xml`
   без проверки успеха. При провале (п.1) на устройстве оставался **прежний** `ui_dump.xml`
   (или локальная копия), и `pull`+`read` возвращали СТАРЫЙ снимок UI. Отсюда «видит не тот экран»
   и «landscape-координаты в портрете» (старый дамп был снят в другой ориентации).

> Первоначальная гипотеза (баг в ротации uiautomator) оказалась НЕверной — это был просто старый
> дамп, снятый когда устройство держали в другой ориентации (приложение `fullSensor`).

## Почему всплыло именно на Settings/standby

Раньше дампы шли по статичным экранам (диалоги, формы) и при подключённой камере (TextureView не
ломает idle). А standby-экран с бесконечной Compose-анимацией idle не достигает никогда → дамп падал.

## Фикс (`tools/ui.mjs`)

1. **Свежесть гарантирована удалением старого файла.** `dumpUi()` сначала `rm -f` remote и обнуляет
   local. Любой валидный файл, который удалось `pull` после этого, — заведомо ТЕКУЩИЙ. Это надёжнее
   парсинга вывода (ошибка idle уходит в stderr, который `execSync` не возвращает).
2. **Авто-отключение анимаций + ретрай.** При провале дампа инструмент один раз ставит системные
   `window/transition/animator_animation_scale = 0` (Compose уважает `animator_duration_scale` →
   анимации замирают → UI достигает idle), ждёт 1.5 с и повторяет. До 5 попыток.
3. **Команда `ui.mjs anim [on|off]`** — ручное управление анимациями (off — для дампов, on — вернуть).
4. **Заголовок ориентации в `dump`**: `📐 screen WxH · portrait/landscape — coords valid for THIS
   orientation only`. Приложение `fullSensor` крутится за устройством — координаты валидны только для
   ориентации на момент дампа. Правило: пользуйся атомарным `ui.mjs tap <query>` (dump+tap в одном
   вызове), не таскай координаты между отдельными вызовами, держи устройство неподвижно при тестах.

## Проверка

- `anim on` → `ui.mjs dump`: первая попытка падает (idle), инструмент сам выключает анимации,
  повторяет и отдаёт СВЕЖИЙ standby-экран (1600×2560). ✅
- `tap "menu" → "settings" → "author"` атомарно находят и жмут элементы. ✅

## Уроки

1. **uiautomator dump требует idle — это его тупое ограничение** (экран вправе анимироваться).
   Любой инструмент поверх него обязан: удалять старый дамп заранее, проверять свежесть, и гасить
   анимации при провале.
2. **stderr ≠ stdout**: `execSync` не возвращает stderr. Не полагайся на текст ошибки uiautomator —
   проверяй факт создания свежего файла.
3. Приложение `fullSensor` + ручное устройство = ориентация меняется между вызовами. Атомарный
   `tap`, а скриншоты — только для визуальной сверки.
