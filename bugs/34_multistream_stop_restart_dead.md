# Bug 34 — Мультистрим: после стопа второй Go Live мёртв; падение одной платформы гасит все выходы

> Заведён: 2026-07-12 (ревизия кода, аудит streaming). Приоритет: **КРИТИЧЕСКИЙ** —
> мультистрим (plans/07 S1-S4) помечен «движок готов», но в бою он сломан.
> Найден статическим аудитом + дизассемблированием RootEncoder 2.4.7 (javap из Gradle-кеша).
> Полный отчёт: `researches/code_audit_2026-07-12.md`.

## Симптом (ожидаемый в бою)

1. Go Live → стрим идёт → Stop → **повторный Go Live вечно висит в Connecting** (ни одного коллбэка)
   до полного перезапуска приложения.
2. При стриме на 2+ платформы: **упала одна (кривой ключ/сеть) → гаснут ВСЕ** — и восстановиться
   нельзя (см. п.1). Упал Instagram → умер и YouTube.
3. Быстрый провал соединения → чёрный экран превью (не восстанавливается).

## Root cause (подтверждено байткодом RootEncoder 2.4.7)

- **`MultiStream.rtpStopStream()` — ПУСТОЙ метод** (`Code: 0: return`). Наш
  `RtmpStreamer.stopStream()` (`RtmpStreamer.kt:570-579`) зовёт no-arg `StreamBase.stopStream()`,
  который делегирует в этот пустой метод → **`RtmpClient.disconnect()` не вызывается нигде**:
  - TCP-сокеты платформ остаются открытыми, `RtmpClient.isStreaming` остаётся true;
  - следующий `MultiStream.startStream(type,i,url)` видит streaming-клиента →
    `shouldStartEncoder=false` → энкодер не стартует;
  - `RtmpClient.connect()` при `isStreaming=true` — no-op. Итог: второй эфир мёртв.
  - Комментарий в коде («гасит ВСЕ выходы + энкодер») — неверен.
- **`onConnectionFailed`** (`RtmpStreamer.kt:209-214`) на сбой ОДНОЙ платформы зовёт общий
  `stopStream()` → энкодер гаснет для всех; плюс невосстановимое состояние из первого пункта.
- **Один общий ConnectChecker на все 4 слота** (`RtmpStreamer.kt:197-248`) — событ< я не
  атрибутируются к выходу: нельзя узнать, КАКАЯ платформа упала; `Live` объявляется по первому
  успеху; `bitrateKbps` мигает между выходами.
- **Частичный старт не обработан** (`RtmpStreamer.kt:548-567`): бросок на i=1 оставляет выход 0
  в эфире при UI-состоянии Error.
- **Реконнекта нет вообще** (`onDisconnect` → Error, `shouldRetry`/`reTry` библиотеки не
  используются) — секундный сетевой блип = конец эфира.
- **Превью после быстрого failed** не восстанавливается (нет `startPreview` в onConnectionFailed,
  в отличие от всех остальных error-путей: 543, 565, 716, 729).

## Как воспроизвести (минимальная приёмка мультистрима!)

1. Один профиль включён → Go Live → Stop → Go Live снова. Ожидание: эфир стартует. Факт: Connecting навсегда.
2. Два профиля (один с заведомо кривым URL) → Go Live. Ожидание: живой стримит, кривой показывает
   ошибку. Факт: гаснут оба.
   Полигон: `node tools/rtmp-server.mjs start` (локальный MediaMTX) — живой выход; кривой URL —
   `rtmp://127.0.0.1:9/dead`.

## План фикса → **plans/09_multistream_stabilization.md** (S1-S5)

Кратко: per-index `stopStream(MultiType.RTMP, i)` по `activeRtmpOutputs` + затем no-arg (энкодер);
per-index ConnectChecker'ы (`Array { i -> makeChecker(i) }`) + per-output state; в onConnectionFailed
стопить только упавший индекс; откат частичного старта; реконнект с бэкоффом через
`getStreamClient(RTMP,i)`/`shouldRetry`/`reTry`; восстановление превью в failed-пути.
