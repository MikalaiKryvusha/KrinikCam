# Bug 45 — Запись видео в файл стартует с большой задержкой (режет начало фразы)

> Заведён: 2026-07-18 (наблюдение Криника при живой записи). Приоритет: ВЫСОКИЙ для UX —
> «думал, запись уже идёт, а начало моей фразы отрезало».

## Симптом

Криник нажал «Запись», бейдж/индикатор показал, что запись идёт — но по факту файл начал писаться с
заметной задержкой, и **начало произнесённой фразы в записи отсутствует**. То есть индикация «идёт
запись» появляется РАНЬШЕ, чем реально пишутся кадры/звук.

## Репро (детерминированное)

1. `stream-to-file on`, затем нажать «Запись» (или харнес `go-live`) и СРАЗУ говорить/хлопнуть.
2. Остановить, вытащить mp4, посмотреть первые секунды: начало обрезано / индикатор был «активен»
   до появления первого кадра.
3. Замер: сравнить timestamp «показали LIVE-бейдж» (`_state.value = Live()`) с первым реально
   записанным кадром (RecordController → RECORDING / первый муксенный сэмпл).

## Root cause (гипотезы)

В `RtmpStreamer.startRecordToFile` индикация выставляется НЕМЕДЛЕННО:
`_state.value = StreamState.Live()` идёт СРАЗУ после `stream.startRecord(path)`, но до этого момента
успевает пройти дорогая настройка: `stopPreview()` → `configureCaptureRotation` (**рестарт
video-source `changeVideoSource` под новую геометрию энкодера**) → `prepareVideo` + `prepareAudio` →
`startRecord` → и только потом GL-пайплайн реально прогревается (`schedulePreviewRestoreAfterStream`
ждёт `gl.isRunning` до 5с). Плюс MediaCodec/аудио-record стартуют не мгновенно (в логах приёмки было
видно `AudioRecord: dead IAudioRecord ... creating a new one` — переинициализация записи звука).
Итог: MediaMuxer получает первый семпл спустя сотни мс–секунду после того, как юзер уже говорит.

## План фикса

> Детальный план исполнения → **`plans/15_record_start_latency.md`**. Ниже — краткая версия.

1. **Замер сначала** (BUG_FIXING: сначала воспроизвести и измерить): залогировать Δ между `startRecord`
   и первым `onStatusChange(RECORDING)` / первым видеокадром в муксер; понять, где основная задержка
   (source-restart? аудио-init? GL-прогрев?).
2. **Индикация по факту, а не по намерению:** переключать UI в «идёт запись» ТОЛЬКО по
   `RecordController.Status.RECORDING` (первый реальный семпл), а до этого показывать «подготовка…».
   Тогда юзер не заговорит в пустоту. (Дешёвый, честный фикс — точно входит.)
3. **Сократить задержку старта:** если основной вклад — рестарт video-source в `configureCaptureRotation`
   при записи (геометрия энкодера обычно = превью), пропускать `changeVideoSource`, когда размеры не
   поменялись. Возможно держать энкодер «тёплым»/пред-прогревать GL. Аккуратно (не сломать поворот).
4. **Приёмка:** записать с хлопком в момент старта → первый кадр/звук на месте (хлопок слышен/виден в
   начале); замер Δ старта до и после (должен упасть, и/или индикатор совпадать с реальным началом).

## Связи

Тот же путь записи, что idea 10/11/17. Не путать с адаптером битрейта (idea 37) — там при записи
target=0, адаптер выключен.
