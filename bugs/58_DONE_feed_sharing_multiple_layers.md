# Bug 58 — Один и тот же фид на двух слоях → креш/зависание

> Заведён: 2026-07-18 (Криник, приёмка мульти-источников 0.7 46). Приоритет: ВЫСОКИЙ (краш).
> Родственные: idea 21 Фаза B (мульти-источники), bug 57/59.

## Симптом

Нельзя назначить ОДИН И ТОТ ЖЕ источник (напр. одну вебку или одну фронталку) двум разным слоям-
камерам одновременно — приложение крешится / зависает.

## Репро

1. Слой «camera» = вебка (UVC).
2. Добавить camera_1, назначить ему ТУ ЖЕ вебку (UVC) — либо обоим фронталку.
3. Креш / зависание.

## Root cause

Физическая камера (одно USB-UVC устройство / один Camera2 cameraId) может быть ОТКРЫТА ТОЛЬКО ОДНИМ
продюсером одновременно. Мульти-источники дают КАЖДОМУ слою свой продюсер+поверхность (CameraSlot).
Два продюсера на одно устройство:
- **UVC:** оба слоя берут один и тот же `usbState.activeCamera` (AUSBC-объект) и зовут `openCamera` в
  РАЗНЫЕ SurfaceTexture → нативный конфликт AUSBC (устройство уже открыто / гонка BufferQueue) → SIGABRT/hang.
- **Camera2 (фронталка):** `DeviceCameraOpener` открывает cameraId; второй `openCamera` того же id →
  `CameraAccessException`/`ERROR_CAMERA_IN_USE` или зависание.

Один продюсер физически пишет в ОДНУ поверхность — «раздать» его в две OES напрямую нельзя.

## План фикса

Варианты (выбрать):
1. **Запрет дубликата (просто, надёжно):** валидация при выборе источника — если источник уже занят
   ДРУГИМ слоем, либо (а) не давать выбрать (серым/тост «источник уже используется слоем X»), либо
   (б) авто-снять его с того слоя (переставить). Крешей нет.
2. **Шаринг фида (сложнее, лучше UX):** один продюсер на физ-устройство; его кадр раздавать в НЕСКОЛЬКО
   слоёв-квадов (как было в единой камере — рисовать один OES в двух слоях). Требует: композитор
   маппит НЕСКОЛЬКО Camera-слоёв на ОДИН CameraSlot по физ-источнику (не по layerId), либо копирует
   снапшот слота в несколько выводов. Тогда «одна камера в двух местах» (напр. основная + зеркальный
   PiP) работает без второго продюсера.

Рекомендация: сначала Вариант 1 (убрать краш валидацией), Вариант 2 — если Криник захочет шаринг.

## Фикс (2026-07-19) — ВАРИАНТ 2 (ШАРИНГ ФИДА), по решению Криника

Криник: «запрет — костыль; хочу, чтобы один фид можно было класть на несколько слоёв». Сделано как в
OBS «дублировать источник»: физическое устройство открывается ОДИН раз, его кадр раздаётся в несколько
слоёв-квадов (каждый со своей позицией/масштабом/поворотом). Второго open того же устройства нет —
краш исчезает как класс.

Механика — «зеркальные слои» (минимальная правка ядра):
- `CameraOpener.sourceKey` — стабильный ключ физ-устройства (`uvc:<id>` / `builtin:<cameraId>`; null =
  виртуалка/нет источника, шаринг не нужен). Файлы: `CameraLayerOpeners.kt`, `DeviceCamera.kt`, `MainScreen.kt`.
- `RtmpStreamer`: трекает `layerSourceKeys[layerId]` из опенеров; `applySceneLayers` группирует слои-
  камеры по ключу — ПЕРВЫЙ в порядке сцены = первичный (держит продюсера), остальные с тем же ключом =
  ЗЕРКАЛА (`CompositorLayer.Camera.mirrorOf = первичный`). Открываем ТОЛЬКО первичный (зеркало продюсера
  не держит) — второго open нет.
- `CompositorVideoSource`: слот (OES/продюсер/снапшот/заглушка) создаётся только для первичных;
  зеркало в `drawFrame`/`layerMatrixOf` рисует слот первичного (`mirrorOf ?: id`) СВОЕЙ трансформой.
- UI-дизейбл «занято» (из промежуточного Варианта 1) СНЯТ — источник теперь выбирается свободно на
  сколько угодно слоёв.

## Приёмка (живьём, полигон @192.168.1.3, 0.7)
- «camera»=фронталка + `camera_1`=та же фронталка, `camera_1`→PiP правый-низ → на превью фронталка
  full-frame И PiP-копия ОДНОВРЕМЕННО, оба живые; отказа в логе НЕТ; PID жив — **краша НЕТ** (скрин `share1`).
- Живость зеркала объективно: 2 кадра через 1,2с, PSNR области PiP ≈ 30 dB (конечный → есть движение,
  не замёрзший; замёрзший дал бы inf).
- Регрессия независимости: `camera_1`→Виртуальная (другой ключ) → PiP стал ТЕСТ-ПАТТЕРНОМ, а не селфи
  (скрин `indep`). Разные источники остаются независимыми (мульти-источники idea 21 целы).
