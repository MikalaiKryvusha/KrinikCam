# Bug 63 — Краш на СТОПЕ трансляции после серии старт-стопов (2 сервиса YouTube+Instagram, 2 слоя)

> Заведён: 2026-07-19 (Криник, живой тест). Устройство: Titan 1, `com.kriniks.kcam.debug` 0.8 (4)-debug.
> Приоритет: ВЫСОКИЙ — краш приложения в бою (мультистрим).

## Симптом

После нескольких последовательных **старт-стоп** трансляций с двумя настроенными выходами
(**YouTube + Instagram**) и двумя слоями «Источник видео» (**основная камера** — сломана, нет видео,
bug 60 + **селфи** — видео есть) → на **очередном СТОПЕ** трансляции приложение **падает**.

Накопительный характер: краш не на первом стопе, а на N-м цикле go-live→stop.

## Репро

1. Настроить 2 выхода: **YouTube** + **Instagram**.
2. Сцена: 2 слоя «Источник видео» — основная камера (без видео, bug 60) + селфи (видео).
3. Go Live → Stop, повторить несколько раз подряд.
4. На очередном стопе — **краш**.

## Форензика

- На момент осмотра: `adb logcat -b crash -d` **пуст** (буфер перекрутился), `/data/tombstones`
  недоступны без root. **Стектрейс не снят.**
- ⚠️ Root cause БЕЗ стектрейса НЕ утверждать (правило BUG_FIXING). Нужен свежий repro с live logcat в
  момент краша.

## Root cause — ГИПОТЕЗЫ (нужен стектрейс)

Накопительный краш на стопе после серии циклов = что-то не освобождается между стопами. Кандидаты
(семьи известных крашей):
- **bug 34** (мультистрим: стоп второго go-live мёртв / падение одного выхода гасит все) — 2 выхода
  (YouTube+Instagram) на стопе гоняют teardown двух `RtmpClient`/энкодеров.
- **bug 48 / EGL_BAD_SURFACE в HWUI** — churn превью-поверхности на стопе (`stopPreview` +
  preview-restore) накапливается за 2-3 цикла → SIGABRT в RenderThread. Тот же паттерн, что и здесь
  (краш на N-м цикле).
- **bug 20 / 27 / 31** — семья EGL-крашей на пересоздании контекста/поверхности.
- Возможный вклад **сломанной основной камеры** (bug 60): молчащий источник + standby в сцене может
  влиять на порядок teardown на стопе.

## План фикса (первый шаг — форензика, НЕ вслепую)

1. **Свежий repro** на девайсе/харнесе: `adb logcat` (весь буфер) во время серии go-live→stop с 2
   выходами + 2 слоя → поймать FATAL / нативный tombstone. Определить: нативный SIGABRT (EGL/HWUI) vs
   Kotlin exception, и в каком компоненте (стоп мультивыхода / teardown поверхности).
2. По стектрейсу — точечный фикс в русле bug 34 / bug 48 (аккуратно, GL/поверхности — зона слепых
   регрессий).
3. **Приёмка:** 10× go-live→stop подряд с 2 выходами (YouTube+Instagram) + 2 слоя (основная+селфи) →
   0 крашей.

## Связи

- **bug 34** (мультистрим стоп/рестарт, падение выхода), **bug 48** (EGL_BAD_SURFACE на churn при
  стопе), bug 20 / 27 / 31 (семья EGL-крашей), bug 58 (краш при мультислоях).
- bug 60 (сломанная основная камера в этой же сцене).

## ✅ ВОСПРОИЗВЕДЁН + ROOT CAUSE + ФИКС (2026-07-19, беспроводной ADB, Titan 1)

**Repro (детерминированный):** харнес, сцена «селфи + добавленная основная (заблокирована)», серия
`go-live→stop` в режиме записи-в-файл. **Краш стабильно на 2-м цикле стопа.**

**Стектрейс (крэш-буфер, по горячему):**
```
Fatal signal 6 (SIGABRT) в tid RenderThread
Abort message: 'Failed to set damage region on surface 0x…, error=EGL_BAD_SURFACE'
 #04 libhwui EglManager::damageFrame
 #05 libhwui SkiaOpenGLPipeline::draw
 #06 libhwui CanvasContext::draw
 #08 libhwui RenderThread::threadLoop
```

**Root cause:** это **семья bug 48/27/31** — `EGL_BAD_SURFACE` в СИСТЕМНОМ HWUI-рендерпотоке (не в нашем
GL-композиторе). На КАЖДОМ стопе (`stopStream`/`stopRecordToFile`) мы звали `startPreview(tv)`, а он
внутри делает `stopPreview()`+`startPreview()` — пересобирает поверхность TextureView. Но во время
эфира превью УЖЕ было привязано (`schedulePreviewRestoreAfterStream`) → это ЛИШНЯЯ пересборка, и она
гонится с HWUI RenderThread, который в тот же момент рисует ту же поверхность → abort. Накопительно =
краш на 2-3 цикле.

**Фикс:** `RtmpStreamer.restorePreviewIfDetached()` — на стопе превью пере-цепляем ТОЛЬКО если оно
реально отвалилось (`!isOnPreview`). Если живо — поверхность НЕ трогаем, лишь освежаем слои/аспект.
Композитор и так рисует в привязанную поверхность. (Смежно убран трэш камеры — bug 60.)

**Приёмка (харнес):** 10× `go-live→stop` подряд (селфи+основная) → **0 крашей** (было — на 2-м);
крэш-буфер пуст; превью после серии живое (не чёрное). ⚠️ Проверено на записи-в-файл (ОДИН выход):
реальный мультивыход (2 RTMP: YouTube+Instagram) НЕ тестировался (не гоняю эфир на реальные каналы
Криника без спроса) — но корень (churn поверхности превью на стопе) ОБЩИЙ, фикс должен покрыть и его.

## Статус
🟢 ФИКС ВНЕСЁН + harness-проверка (10 циклов, 0 крашей). Ждёт приёмки Криника на РЕАЛЬНОМ мультивыходе
(YouTube+Instagram, 2 слоя). Root cause подтверждён стектрейсом (EGL_BAD_SURFACE в HWUI = семья bug 48).
