# Bug 42 — Нет профиля кодека AV1 — добавить

> Заведён: 2026-07-18 (ревью Криником фичи «профиль кодера»). Приоритет: СРЕДНИЙ.
> Реализуется в менеджере профилей кодера (bug 41).

## Симптом

В выборе видеокодека профиля есть только H.264 и H.265. Криник хочет ещё **AV1**.

## Контекст / выполнимость

Библиотека RootEncoder 2.4.7 УЖЕ умеет AV1: `com.pedro.common.VideoCodec` содержит `H264, H265, AV1`
(сверено `javap`). Агент намеренно не выносил AV1 в UI («сырое покрытие по RTMP-платформам») — но это
решение Криника, не агента. Добавляем.

## Root cause

Домен-enum `VideoCodec` (`data/profiles/.../model/StreamProfile.kt`) объявлен только с H264/H265; в
`RtmpStreamer.toPedro()` маппинг тоже на два значения; в UI дропдаун берёт `VideoCodec.entries`.

## План фикса

> Реализуется в рамках **`plans/14_encoder_profile_manager.md`** (S1 enum `VideoCodec += AV1`, S2
> маппинг + проверка аппаратной поддержки, S3 UI-подсказка).

1. `VideoCodec` (домен): добавить `AV1("AV1")`.
2. `RtmpStreamer.toPedro()`: ветка `VideoCodec.AV1 -> PedroVideoCodec.AV1`.
3. UI-дропдаун кодека подхватит автоматически (`VideoCodec.entries`). Подсказку про совместимость
   расширить: **AV1 по RTMP принимают ЕДИНИЦЫ площадок; для записи в файл — ок** (та же логика, что у
   H.265, только жёстче). Показать предупреждение при выборе AV1.
4. **Приёмка:** запись профиля с AV1 через харнес `go-live 720,AV1,...` → ffprobe: `codec_name=av1`,
   файл валиден (кадры считаются). Проверить, что устройство (Titan 1) вообще имеет AV1-энкодер
   (`MediaCodecList`): если аппаратного нет — prepareVideo вернёт false, показать честный снэкбар
   «AV1 не поддерживается этим устройством» (не молчать — правило idea 20).

> Примечание: делать в рамках менеджера профилей кодера (bug 41), либо отдельным мелким фиксом на
> текущей структуре — по решению Криника, что раньше.

## Статус
🔴 ОТКРЫТ (средний). Библиотека готова; нужно вынести в enum+маппинг+UI и проверить аппаратную поддержку.
