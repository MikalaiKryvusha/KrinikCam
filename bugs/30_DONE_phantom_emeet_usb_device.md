# Bug 30 — Фантомная USB-камера «EMEET Piko+» в системе (не подключена физически)

**Статус:** 🟡 ОТКРЫТ — репорт Криника (2026-07-06). Низкий приоритет (системный артефакт, приложение
не затронуто), но зафиксирован по просьбе.
**Устройство:** Headwolf Titan 1 @ 192.168.50.187.

## Симптом

`adb shell dumpsys usb` показывает ДВЕ USB-камеры:
```
product_name=2K USB Camera      ← реально воткнута
product_name=EMEET Piko+        ← ФАНТОМ: физически НЕ подключена (оставлена дома, нет в здании)
```
Emeet Piko+ никак не подключена, но система держит её в списке USB-устройств.

## Диагностика

- Это **системный кэш Android USB HAL** (stale device record), а НЕ наш код.
- **Приложение KrinikCam её НЕ видит:** в logcat нет attach-события Emeet (`USB attached` / AUSBC
  `UVC connected`) — только реальная 2K. Т.е. AUSBC USBMonitor фантом не подхватывает, активный
  источник фантомом не становится. Приложение работает корректно.
- Риск теоретический: если где-то использовать `UsbManager.getDeviceList()` напрямую (а не
  attach-события) — фантом попадёт в перечисление. Мы так не делаем (идём от attach-событий AUSBC).

## Root cause (гипотеза)

Прошивка Headwolf Titan1 / Android USB-стек не очищает запись об устройстве после физического
отключения (или помнит по последнему подключению). Не воспроизводится в нашем коде — вне нашего
контроля. Лечится обычно перезагрузкой устройства / переподключением USB-хаба.

## План

- [ ] Убедиться, что нигде не используем `UsbManager.getDeviceList()` для авто-открытия камеры (идём
      только от attach-событий AUSBC) — тогда фантом на приложение не влияет. (Проверка кода.)
- [ ] Если понадобится список камер для UI выбора источника (idea 20) — фильтровать по реальному
      `hasPermission`/успешному открытию, не показывать фантомы.
- [ ] Кринику: перезагрузить планшет / переткнуть USB — уйдёт ли фантом (подтвердить, что системный).
- Приоритет низкий: не ломает работу, приложение фантом игнорирует.
