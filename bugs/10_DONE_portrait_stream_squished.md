## 🧠 ПЕРЕОСМЫСЛЕНИЕ (по тесту Криника 0.3(16) + PHILOSOPHY) — 2026-06-29, сессия 2

Тест Криника показал, что стало ХУЖЕ: при старте/остановке стрима превью **накручивает лишние 90°** и
искажается. Причина простая и она же — главный урок: **в коде РАБОТАЮТ ДВА механизма поворота
одновременно, и они дерутся:**
1. **Поворот ПРЕВЬЮ** — матрица TextureView (`UvcPreviewView.applyPreviewRotation`). Без стрима крутит
   ИДЕАЛЬНО (Криник подтвердил).
2. **Поворот ЭНКОДЕРА/источника** — при старте стрима меняются размеры канваса + (variant C) крутит
   источник. Плюс код переключает матрицу превью (rotationDegrees→0 в эфире).
Итог: повороты складываются/накапливаются → «полный ужас».

Корень дисторсии (декомпиляция `SizeCalculator.getScale`, подтверждено): **энкодерный поворот
библиотеки даёт неравномерный scale, заточенный под камеру** → для нашего Surface-источника искажает.

### ✅ ЕДИНЫЙ ПРОСТОЙ ПЛАН (ОДИН поворот в ОДНОМ месте — по модели Криника)

Источник ВСЕГДА получает 16:9 landscape (факт UVC). Мы поворачиваем его ОДИН раз — **в самом
источнике** (variant C) — и этот готовый кадр идёт И в превью, И в энкодер ОДИНАКОВО. «Что видишь = то
и стримишь». Энкодер: `rotation=0`, размеры = выход источника (1080×1920 для 90/270) → `getScale`=(1,1),
без искажений. **Матрицу поворота превью УБРАТЬ полностью** — она и есть второй, дерущийся поворот.

Что сделать (следующий заход, аккуратно, не в спешке):
1. `UvcPreviewView`: убрать матричный поворот (rotationDegrees → всегда identity). Превью просто
   показывает кадр источника (портрет леттербоксится во вью — это нормально).
2. `RtmpStreamer`: ОДНО место вычисляет выходные размеры из `_videoRotation` (портрет для 90/270),
   используется И для превью-канваса, И для `prepareVideo`. При смене угла (розовая кнопка) в idle —
   переинициализировать GL/encoder size + **перезапустить источник** с новым буфером (setDefaultBufferSize)
   и `setOutputRotation`. **КЛЮЧ:** буфер источника обязан совпадать с размером энкодера и с
   outputRotation — иначе in-source поворот мажет (это и есть причина регрессии в 0.3(16): источник
   не перезапустился с портретным буфером).
3. `startStream`/`startRecordToFile`: `prepareVideo(outputDims, rotation=0)` + `setOutputRotation(deg)`.
   УБРАТЬ `setStreamRotation`/`(360-x)`-инверсию/жонглирование `setCameraOrientation` и переключение
   матрицы превью на старте/стопе стрима.
4. Источник (`VirtualVideoSource` + `UvcVideoSource`) чисто перезапускается с портретным буфером.
5. Проверка: harness (запись = круглый круг, заполняет) + Криник смотрит живое превью на консистентность.

> ⚠️ Текущий незакоммиченный код (variant C) — НЕЗАВЕРШЁННЫЙ шаг: источник крутит, но НЕ
> перезапускается с портретным буфером, и матрица превью НЕ убрана → дерётся. Доделать по плану выше.

# БАЗА ЗНАНИЙ — сырые данные гуглёжа (2026-06-29)

## A. AspectRatioMode — что реально делает (discussion #1546, ответ мейнтейнера pedroSG94)

Дословно (перевод сохраняет смысл):
- **Adjust** = «Adapt image to full size of the preview view. If the preview have a different aspect
  ratio then the preview is filled with black gutters to avoid distortion.» = Android `centerInside`
  (леттербокс чёрными полями, без искажения).
- **Fill** = «...image cropped to avoid distortion.» = `centerCrop` (кроп, без искажения).
- **None** = «Fill the image with the preview, doing distortion if necessary.» = `fitXY` (растягивает
  с искажением).
- ⚠️ Мейнтейнер: aspect-ratio корректировки применяются ТОЛЬКО когда view — это `OpenGlView`. Если
  передать обычный `SurfaceView` / кастомную поверхность, библиотека НЕ применяет aspect-настройки.
- ⚠️ В discussion НЕТ гайда по портрету/9:16, setEncoderSize, rotation — только определения режимов.
- Источник: https://github.com/pedroSG94/RootEncoder/discussions/1546

## B. OpenGL-пайплайн: encoder и preview РАЗДЕЛЬНЫ (DeepWiki RootEncoder, 3.4 OpenGL Rendering Pipeline)

Дословные сигнатуры рендера (ключевое!):
- Encoder: `mainRender.drawScreenEncoder(encoderWidth, encoderHeight, isPortrait, streamOrientation,
  isStreamVerticalFlip, isStreamHorizontalFlip, streamViewPort)` → вьюпорт считает
  `SizeCalculator.calculateViewPortEncoder()`.
- Preview: `mainRender.drawScreenPreview(previewWidth, previewHeight, isPortraitPreview,
  **aspectRatioMode**, previewOrientation, ...)`.
- **🔑 У encoder-отрисовки в сигнатуре НЕТ `aspectRatioMode` — он есть ТОЛЬКО у preview!** То есть
  `setAspectRatioMode(Adjust)` влияет на ПРЕВЬЮ, а энкодер считает вьюпорт иначе. Это объясняет, почему
  превью у нас леттербоксится корректно (через матрицу + Adjust), а ЭНКОДЕР сжимает.
- Поворот раздельный: encoder → `streamOrientation` (`setStreamRotation`); preview →
  `previewOrientation` (`setPreviewRotation`); вход камеры → `CameraRender` («applies rotation and
  flip transformations to the input») управляется `setCameraOrientation`.
- `setEncoderSize(w,h)` — разрешение стрима; `setEncoderRecordSize` — для записи (другое разрешение).
- VideoSource-кадр идёт через `CameraRender` (OES→GL_TEXTURE_2D), потом `ScreenRender` применяет
  трансформации под целевую поверхность.
- «Each surface independently calculates viewports based on its dimensions and aspect mode — no
  shared calculations between preview and encoder.»
- Источник: https://deepwiki.com/pedroSG94/RootEncoder/3.4-opengl-rendering-pipeline

# АНАЛИЗ КОДА — где может теряться ориентация/аспект (без фикса, только разбор)

## Цепочка кадра (наша архитектура)

```
USB-камера (AUSBC, 1920×1080 landscape)
  → UvcVideoSource.start(surfaceTexture)         [:app/streaming/UvcVideoSource.kt]
     openCamera(surfaceTexture, previewW=1920, previewH=1080)   ← всегда 1920×1080!
  → GL input SurfaceTexture (RootEncoder)
  → CameraRender (поворот/флип входа ← setCameraOrientation)
  → drawScreenEncoder(encoderW, encoderH, isPortrait, streamOrientation)  → MediaCodec → RTMP
  → drawScreenPreview(previewW, previewH, aspectRatioMode, previewOrientation) → TextureView (превью)
```

## 🔑 Главное подозрение: `UvcVideoSource` ИГНОРИРУЕТ параметры от RootEncoder

`UvcVideoSource.create(width, height, fps, rotation): Boolean = true` — мы **игнорируем** width/height/
rotation, которые RootEncoder передаёт (это размер/поворот ЭНКОДЕРА). А в `start()` камера всегда
открывается на **1920×1080 landscape** (`previewWidth/Height`), независимо от того, портретный энкодер
или нет.

Следствия (гипотезы для проверки в focused-сессии):
1. Когда энкодер портретный (1080×1920), RootEncoder зовёт `create(1080,1920,…)`, но источник всё
   равно даёт 1920×1080 → GL обязан вписать 16:9-текстуру в 9:16-канвас. А `calculateViewPortEncoder`
   (без aspectRatioMode) при этом, видимо, делает `fitXY` (растягивает) → сжатие/растяжение.
2. `SurfaceTexture.setDefaultBufferSize` — RootEncoder обычно ставит его по размерам из `create()`
   ДО `start()`. Но AUSBC внутри `openCamera` может переопределить буфер под свой preview size
   (1920×1080). Рассинхрон размера буфера источника и канваса энкодера = масштабирование.
3. `previewWidth/previewHeight` захардкожены 1920×1080. Для портрета источник стоило бы открывать
   так, чтобы его кадр геометрически ложился в 9:16 (поворот в самом источнике/буфере), а не
   полагаться на encoder-вьюпорт, который аспект не держит.

## 📚 БАЗА ЗНАНИЙ — исследование без кода (/bug-research) — 2026-06-29

### Сырые данные: как RootEncoder делает поворот ШТАТНО

Канонический пример из репозитория (`app/.../rotation/CameraFragment.kt`):
```kotlin
val width = 640
val height = 480
private var rotation = 0
fun setOrientationMode(isVertical: Boolean) {
    rotation = if (isVertical) 90 else 0   // вертикаль = 90, НЕ 270, НЕ swap размеров
    prepare()
}
// prepare():
genericStream.prepareVideo(width, height, vBitrate, rotation = rotation)
```
Документация (DeepWiki/wiki): *«Orientation vertical means stream with vertical resolution (width=640,
height=480 and rotation=90). The stream/record result will be 480x640 resolution.»*
→ **W/H НЕ меняешь руками; передаёшь размеры источника + `rotation`; библиотека сама даёт повёрнутый
выход и сохраняет пропорции.** Источники:
- https://github.com/pedroSG94/RootEncoder (раздел про rotation/orientation)
- https://github.com/pedroSG94/RootEncoder/issues/531 , #499, #429 (rotation consistency)
- пример: `app/src/main/java/com/pedro/streamer/rotation/{CameraFragment,RotationActivity}.kt`

### Сырые данные: декомпиляция `library-2.4.7.aar` / `encoder-2.4.7.aar` (javap)

- `StreamBase`: `prepareVideo(int w, int h, int bitrate, int fps, int iframe, int rotation)` —
  rotation это просто 6-й аргумент. Также есть `setOrientation(int)`.
- `GlStreamInterface`: поля `streamOrientation`, `previewOrientation`, `orientationForced`
  (enum PORTRAIT/LANDSCAPE/NONE), `aspectRatioMode`, `autoHandleOrientation`. Методы
  `setStreamRotation(int)`, `setRotation(int)`, `setCameraOrientation(int)`, `setPreviewRotation(int)`,
  `setAspectRatioMode(...)`, `forceOrientation(...)`. → `rotation` в prepareVideo, `setStreamRotation`
  и `setRotation` пишут в ОДНО поле `streamOrientation` — это ОДИН механизм, не три разных.
- `MainRender`: `drawScreenEncoder(int w, int h, boolean, int rotation, boolean, boolean)` — **БЕЗ
  AspectRatioMode** (энкодер заполняет/fill). `drawScreenPreview(int, int, boolean, AspectRatioMode,
  int rotation, ...)` — **С AspectRatioMode** (превью леттербоксит). → preview и encoder вписывают
  по-разному by design: encoder = fill, preview = letterbox/adjust.

### 🎯 ROOT CAUSE (гипотеза, обоснованная)

Мы **боролись с библиотекой и складывали несколько поворотов сразу** (ровно то, о чём PHILOSOPHY):
1. Инвертировали угол: `streamRotation = (360 - ui) % 360` → для ui=90 слали **270** вместо 90 →
   контент «вверх ногами».
2. ПОСЛЕ prepareVideo звали `applyVideoRotation()` (+ `setCameraOrientation`/`setStreamRotation`) →
   вторая трансформация поверх первой → **растяжение (овал)**.

Энкодер заполняет сцену в размеры выхода. Это НЕ баг: при `rotation=90` выход = 1080×1920, а
повёрнутый 16:9-источник имеет отпечаток 9:16 = аспект выхода → заполняет БЕЗ растяжения (как 4:3→3:4
в примере даёт чистый 480×640). Дисторсия у нас — ТОЛЬКО от двойной трансформации и инверсии угла,
а НЕ от отсутствия AspectRatioMode у энкодера.

### 🎯🎯 ОКОНЧАТЕЛЬНЫЙ ROOT CAUSE — декомпиляция `SizeCalculator` (2026-06-29, сессия 2)

После ~5 эмпирических попыток (все дали искажение — подтверждено Криником глазами) раскодирован
байткод `com.pedro.encoder.utils.gl.SizeCalculator`:
- `MainRender.drawScreenEncoder` строит матрицу через `processMatrix(rotation, flipH, flipV, m)`,
  которая зовёт `getScale(rotation, w, h, isPortrait, …)`.
- **`getScale` при rotation 90/270 возвращает НЕРАВНОМЕРНЫЙ масштаб** (sx≠sy: умножает одну ось на
  ≈`(w/h)²`). Именно это и есть растяжение/овал. Формула рассчитана на КАМЕРУ (аспект сенсора связан с
  аспектом энкодера определённым образом). Наш источник рисует 16:9 в Surface НАПРЯМУЮ (не через
  CameraRender) → в формулу не вписывается → энкодерный поворот всегда искажает.
- `calculateViewPortEncoder(w, h, isPortrait)` дополнительно может леттербоксить (отсюда чёрные полосы).

**Подсказка Криника (ключевая):** при 90° ПРЕВЬЮ поворачивается корректно (display-матрица
`UvcPreviewView`), но при СТАРТЕ СТРИМА картинка «отыгрывает назад» → поворот превью и поворот энкодера
— разные механизмы; энкодерный для нашего источника ломается (getScale).

**ВЫВОД:** библиотечный поворот ЭНКОДЕРА (`prepareVideo(rotation)` / `setStreamRotation` / isPortrait)
для нашего НЕ-камерного источника принципиально даёт неравномерный scale. Бороться с ним = усложнять.

### 🔗 СВЕДЕНИЕ старого (разделы A–D, «АНАЛИЗ КОДА») и нового исследования — ИТОГ

Сегодняшний поспешный вывод («дисторсия ТОЛЬКО от стопки трансформаций») — НЕПОЛНЫЙ. Старая
**попытка №4** = чистый `prepareVideo(1920,1080, rotation=90)` без ручного swap — **уже тянула**. Значит

2. **🔑 `UvcVideoSource.create(w,h,rotation)` ИГНОРИРУЕТ параметры** и всегда отдаёт фикс 1920×1080.
   Канонический пример работает, т.к. его камера-источник подстраивается под запрошенную геометрию
   (для 90 даёт 480×640-совместимый кадр). У нас источник = жёсткий 16:9 → энкодер обязан вписывать
   16:9 в 9:16-канвас, без aspect-режима → растяжение или (в exp#2) леттербокс.

**ЕДИНЫЙ ROOT CAUSE (надёжный):** проблема НЕ в выборе rotation-метода, а в том, что **источник
(`UvcVideoSource`/`VirtualVideoSource`) отдаёт фиксированный 16:9-кадр и игнорирует запрошенную
энкодером геометрию**, а наша поверхность — не `OpenGlView`, поэтому библиотека сама аспект не держит.
Поэтому ЛЮБОЙ rotation-метод приводит либо к fitXY-растяжению, либо к letterbox — но НЕ к жёсткому
геометрическому повороту-заполнению, который описывает Криник (повернуть 1920×1080 как есть → 1080×1920,
заполняет, круг круглый).

**ПЛАН (приоритет — простой штатный путь, по PHILOSOPHY):**
- **(A) Сначала попробовать НЕтронутые высокоуровневые API** (мы их НИ РАЗУ не звали; вероятно это и
  есть «всё придумано за нас»): `StreamBase.setOrientation(int)`, `GlStreamInterface.forceOrientation(
  OrientationForced.PORTRAIT)`, `setAutoHandleOrientation(true)`. Возможно они и делают честный
  геометрический поворот-заполнение. Проверить харнессом (круг круглый + заполняет, без полос).
- **(B) Если не выйдет — чинить ИСТОЧНИК (🔑):** заставить `UvcVideoSource.create()`/`VirtualVideoSource`
  уважать переданные w/h/rotation (и/или `surfaceTexture.setDefaultBufferSize(reqW,reqH)`), чтобы
  повёрнутый кадр ложился 1:1 в 9:16-канвас → жёсткий поворот-заполнение.
- **(C) Крайний вариант** — рисовать УЖЕ повёрнутый кадр прямо в источнике (geom-поворот текстурных
  координат на 90° в `VirtualVideoSource`/`UvcVideoSource`).
- НЕ начинать с низкоуровневого комбинирования setStreamRotation/setCameraOrientation/swap — это и
  плодило сложность. Один механизм за раз, проверка формой (круг) через харнесс ~30с.

### ✅ ИСКАЖЕНИЕ ПОБЕЖДЕНО — эксперимент №2 (проверен на девайсе через харнесс) — 2026-06-29

`RtmpStreamer.startRecordToFile`: для 90/270 канвас энкодера делаем **портретным (swap W↔H → 1080×1920)**
+ `setCameraOrientation(0)` + `setStreamRotation(deg)`. Запись при 90°:
- `ffprobe` → **1080×1920** ✅
- кадр (`tools/screenshots/rec90_exp2_frame.jpg`): **круг КРУГЛЫЙ**, сетка квадратная (НЕТ овала),
  ориентация верная (TOP вправо, не вверх ногами). **Дисторсия устранена.**

**Корневое понимание (подтверждено 2 экспериментами):** энкодер ВСЕГДА fitXY-вписывает GL-сцену в
канвас. Значит аспект канваса ДОЛЖЕН совпадать с аспектом повёрнутого контента: 90/270 → 1080×1920.
Тогда поворот через `setStreamRotation` геометрически честный, без сжатия. (Exp#1: landscape-канвас +
setStreamRotation → ориентация верная, но горизонтальный овал, т.к. 9:16-контент растянули на 16:9.)

**Остался ПРОДУКТОВЫЙ вопрос (решение Криника):** 16:9-кадр вписан в 9:16 с **letterbox (чёрные полосы
сверху/снизу)** — виден весь кадр, но не на весь экран. Альтернатива — **fill/crop** (заполнить 9:16,
обрезав длинные края). Превью при 90° заполняет экран сильнее, чем запись → возможна рассинхронизация
preview/encoder. Дальше: (1) выбрать letterbox vs fill, (2) перенести рабочий подход в `startStream`.
