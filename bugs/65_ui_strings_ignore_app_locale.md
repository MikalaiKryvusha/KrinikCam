# Bug 65 — Часть UI не слушается языковой настройки приложения (глобальная локализация)

> Заведён: 2026-07-19 (Криник, живой тест). Устройство: Titan 1, `com.kriniks.kcam.debug` 0.8 (4)-debug.
> Приоритет: СРЕДНИЙ (качество/бренд, широкий охват). Глобальный проход по UI.

## Симптом

Многие элементы UI **не переключаются** на язык, выбранный в Настройках (Settings → Язык приложения,
bug 46). Часть строк остаётся на одном языке / не реагирует на смену глобальной настройки.

## Что нужно (задача)

1. **Пройтись по ВСЕМУ UI** и найти строки, которые не локализуются:
   - хардкод-строки прямо в Composable/диалогах (не через `stringResource`/`strings.xml`);
   - строки, резолвящиеся ОДИН раз на старте (не реагируют на смену локали без рестарта).
2. Вынести хардкод в `strings.xml` (EN дефолт + `values-ru`), заменить на `stringResource`.
3. Убедиться, что смена языка в Настройках **корректно переключает весь UI** согласно глобальной
   настройке приложения (в идеале — без перезапуска; минимум — консистентно после применения).

## Гипотезы / зоны риска (проверить)

- **Хардкод-строки** в композаблах/диалогах/оверлеях (grep по кириллице и англ-литералам в `.kt` UI).
- **Строки, локализуемые на старте:** напр. `DeviceCameraEnumerator.enumerate` формирует имена камер из
  ресурсов ОДИН раз при инициализации (в коде помечено: смена локали отразится после перезапуска —
  пересоздания источников). Аналогичные «снимки» строк.
- **Механизм смены локали (bug 46):** проверить, применяется ли выбранный язык ко всем экранам/оверлеям
  и пересобираются ли строки при смене (recompose vs кэш).
- **contentDescription / динамически собранные строки** (конкатенация в коде вместо форматных ресурсов).

## План (первый шаг — аудит)

1. `grep` по UI-модулям на кириллицу и «голые» строковые литералы в Composable → список хардкода.
2. Проверить путь применения языка (bug 46 / plans/16) — где ставится локаль, что триггерит пересборку.
3. Вынести найденное в ресурсы, заменить на `stringResource`, повторно проверить переключение.
