# Bug 41 — Настройки кодека вшиты в профиль платформы (НЕПРАВИЛЬНО): нужен отдельный менеджер профилей кодера

> Заведён: 2026-07-18 (ревью Криником фичи «профиль кодера»). Приоритет: ВЫСОКИЙ — правит
> архитектуру только что добавленной фичи, пока ей не начали пользоваться (меньше миграций потом).
> Родственные: bug 42 (AV1), bug 43 (битрейт в Мбит), bug 44 (моно) — все реализуются В ЭТОМ менеджере.

## Симптом (что не так)

Агент реализовал «профиль кодера» (разрешение/битрейт/fps/кодек/звук) прямо ВНУТРИ настроек платформы —
поля кодера живут на `StreamProfile` и редактируются в форме платформы (`ProfileEditDialog`). **Криник:
это неправильно.** В настройках платформы должен быть ТОЛЬКО выбор «Профиль кодера» (выпадашка) и всё.
Сам кодек (разрешение, битрейт, fps, видеокодек, звук) настраивается ОТДЕЛЬНО — в отдельном
**менеджере профилей кодера**.

## Почему так правильно (видение Криника)

Один профиль кодера («1080p60 H.264 6 Мбит») переиспользуется на НЕСКОЛЬКИХ платформах — не надо
дублировать одни и те же параметры энкодера в каждой платформе. Платформа = «куда» (URL + ключ +
ссылка на профиль кодера), профиль кодера = «как кодировать». Это разделение ответственности и меньше
дублирования (KISS/Оккам).

## Репро

1. Настройки → Платформы → карандаш на профиле.
2. Наблюдать: в форме платформы прямо тут задаются разрешение, битрейт, кодек, звук.
   Ожидание Криника: тут только поле «Профиль кодера: [выпадашка]» + кнопка «Управление профилями кодера».

## Root cause

Проектное решение агента. При реализации фичи (2026-07-18) агент спросил Криника про ОБЪЁМ (битрейт+звук+
кодек) и UX ВВОДА битрейта, но НЕ про СТРУКТУРУ (отдельная сущность vs поля платформы) — и по умолчанию
положил всё на `StreamProfile`. Урок: структурные продуктовые развилки — через интервью до реализации.

Технически: поля `videoWidth/Height/Fps/BitrateBps/videoCodec/adaptiveBitrate/audio*` сидят на
`StreamProfile` (`data/profiles/.../model/StreamProfile.kt`), схема Room v3.

## План фикса (фазами)

> **Детальный план исполнения → `plans/14_encoder_profile_manager.md`** (этот баг — хребет; bug 42/43/44
> садятся в тот же план). Ниже — краткая версия.

**S1 — модель.** Новая сущность `EncoderProfile` (id, name, videoWidth, videoHeight, videoFps,
videoBitrateBps, videoCodec, adaptiveBitrate, audioBitrateBps, audioSampleRate, audioStereo). Room:
таблица `encoder_profiles` + DAO. `StreamProfile` теряет поля кодера, получает `encoderProfileId: Long`
(ссылка). Миграция Room **v3→v4**: создать таблицу, засеять 1-2 дефолтных профиля кодера
(«1080p30 H.264» и т.п.), перенести значения кодера каждого существующего профиля платформы в новый
профиль кодера и проставить `encoderProfileId`. Дефолтов НЕ терять (правило bug 37: без destructive).

**S2 — репозиторий/резолв.** `ProfilesRepository` отдаёт профили кодера; при старте эфира стример
резолвит `StreamProfile.encoderProfileId → EncoderProfile` и конфигурит энкодер по нему (сейчас берёт
поля прямо у `StreamProfile`). Мультистрим: параметры энкодера — у профиля кодера ПЕРВОГО выхода
(как сейчас у `profiles.first()`).

**S3 — UI менеджера профилей кодера.** Отдельный экран/оверлей: список профилей кодера, CRUD (создать/
править/удалить), импорт/экспорт (как у платформ). Внутри — все поля кодера (сюда переезжают bug 42 AV1,
bug 43 битрейт-Мбит, bug 44 моно). Точка входа — в Settings и/или из формы платформы («Управление…»).

**S4 — форма платформы.** Убрать секции «Видео»/«Звук»; вместо них одна выпадашка «Профиль кодера»
(список профилей кодера) + кнопка перехода в менеджер. Остаётся: имя, платформа, URL, ключ, вкл/выкл,
выбор профиля кодера.

**S5 — приёмка.** Живьём: создать профиль кодера, привязать к 2 платформам, эфир на полигон →
ffprobe (кодек/битрейт/звук из профиля кодера, не из платформы); миграция v3→v4 не теряет данные
(тест как в bug 37/idea 37 — накатить поверх старого билда). Юниты round-trip + резолв FK.

## Решения Криника (2026-07-18) — развилка закрыта

- **Связь = живая ссылка по id, БЕЗ снимков.** Платформа хранит `encoderProfileId`. Отредактировал
  профиль кодера → изменение задевает ВСЕ платформы, настроенные на него. Временных копий-слепков нет.
  → S1: `StreamProfile.encoderProfileId: Long` (FK), при старте эфира резолвим профиль по id (S2).
  Удаление профиля кодера **разрешено ВСЕГДА** (уточнение Криника 2026-07-18): даже если на него
  ссылаются платформы. Ссылавшиеся платформы получают ЗАПАСНОЙ (fallback) профиль — первый оставшийся,
  а если удалили последний, создаётся дефолтный. Инфо-снэкбар «N платформам выдан запасной».
- **Импорт/экспорт (моё предложение по умолчанию, если Криник не против):** экспорт платформы тянет
  за собой её профиль кодера в тот же файл (иначе импортированная платформа ссылается в пустоту);
  при импорте — дедуп профилей кодера по содержимому. Подтвердить при реализации.

## Реализация (2026-07-18, plans/14)
Рефактор СДЕЛАН: отдельная `EncoderProfile` (Room-таблица encoder_profiles, миграция v3→v4),
`StreamProfile.encoderProfileId` (ссылка по id), менеджер `EncoderProfilesOverlay`, форма платформы =
только пикер + «Управление…». Запрет удаления «занятого» профиля (referencingPlatforms). Сборка+юниты
зелёные. **Проверено живьём:** миграция v3→v4 сохранила реальный профиль «YouTube» и создала его профиль
кодера; ffprobe подтвердил, что профиль кодера доходит до энкодера. Детали и остатки — STATUS 2026-07-18.
