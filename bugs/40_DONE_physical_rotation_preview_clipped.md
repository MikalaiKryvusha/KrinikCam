# Bug 40 — Физический поворот планшета: превью «уезжает вниз и обрезается» (регресс no-op фикса bug 27)

> Заведён: 2026-07-12 (Криник, live: «Сейчас виртуалька открыта, планшет в пейзаже. Поворачиваю в
> портрет — уезжает вниз и обрезается за пределы экрана виртуалка»). Приоритет: ВЫСОКИЙ (базовый UX).
> Это та самая живая сверка поворотом, которую ждал bug 27: краша больше НЕТ, но вылез регресс
> глубже — превью не адаптируется к новому размеру окна.

## Симптом

Виртуальная камера открыта, планшет в пейзаже → физический поворот в портрет → композит не
вписывается в новый размер: уезжает вниз, обрезается за пределами экрана. (Ожидаемо симметрично и
для портрет→пейзаж, и для других источников — проверить.)

## Root cause (по байткоду RootEncoder 2.4.7, GlStreamInterface)

Фикс bug 27 (2026-07-07) сделал `onSurfaceTextureSizeChanged` **no-op** (MainScreen:218), предполагая,
что «AspectRatioMode.Adjust вписывает композит в новый размер вью сам». Это НЕВЕРНО:
`GlStreamInterface.draw()` каждый кадр считает вьюпорт превью из **полей `previewWidth`/
`previewHeight`** (передаются в `MainRender.drawScreenPreview(w, h, ...)`), а НЕ из фактического
размера поверхности. Эти поля выставляются только при `startPreview` — после ресайза TextureView они
остаются старыми (ландшафтными) → композит рисуется со старой геометрией в портретное окно →
«уезжает и обрезается».

## Фикс (2026-07-12, версия 0.6 (16))

Обновлять ТОЛЬКО числа вьюпорта, НЕ трогая поверхность (краш-путь bug 27 не задевается):
`onSurfaceTextureSizeChanged → StreamViewModel.onPreviewSizeChanged → repository →
RtmpStreamer.onPreviewSurfaceResized → glInterface.setPreviewResolution(w, h)`
(метод существует в 2.4.7: тупо пишет два поля — байткод; поверхность/EGL не пересоздаются).

Файлы: MainScreen.kt (коллбэк + комментарий), StreamViewModel.kt, StreamingRepository.kt,
RtmpStreamer.kt (`onPreviewSurfaceResized`).

## Приёмка

Живой планшет (руки Криника — ADB-orient НЕ эквивалентен, см. EXPERIENCE EXP-0004):
1. Виртуалка, пейзаж → портрет → композит целиком в экране, леттербокс корректный.
2. Портрет → пейзаж обратно — то же.
3. Несколько циклов подряд — без крашей (bug 27 не регрессировал!) и без «уезжания».
4. То же с UVC-камерой (Piko+).
