# Bug 59 — UVC (Pico+) дублируется в списке источников при вкл/выкл

> Заведён: 2026-07-18 (Криник, приёмка мульти-источников 0.7 46). Приоритет: СРЕДНИЙ.
> Родственные: bug 35 (detach-путь), idea 21 Фаза B (мульти-источники), bug 57/58.

## Симптом

При отключении/подключении Pico+ (UVC) в диалоге выбора источника видео появляется ЕЩЁ ОДНА её копия —
клон в списке. Повторные вкл/выкл множат клоны.

## Репро

1. Открыть выбор источника слоя (⚙ или пикер) — вебка в списке 1 раз.
2. Выдернуть Pico+ → воткнуть.
3. В списке источников появляется ВТОРАЯ «EMEET Piko+» (клон).

## Root cause

`DeviceManager`:
```kotlin
fun notifyUvcConnected(source) { _uvcSources.value = _uvcSources.value + source; ... }   // без дедупа
fun notifyUvcDisconnected(deviceId) { _uvcSources.value = _uvcSources.value.filter { it.id != deviceId } }
```
- `notifyUvcConnected` ДОБАВЛЯЕТ в список без проверки на дубликат.
- При реконнекте USB часто выдаёт НОВЫЙ deviceId (реэнумерация). `notifyUvcDisconnected(старыйId)`
  либо не срабатывает (старый id не совпал / событие detach не дошло — смежно bug 35), либо приходит
  ПОСЛЕ connect. Итог: старая запись (старый id) остаётся + добавляется новая (новый id) → две «Piko+».
- Плюс connect-события AUSBC могут прилетать НЕСКОЛЬКО раз на одно устройство (наблюдали в bug 29.1) →
  тоже дубли.

## План фикса

1. **Дедуп по СТАБИЛЬНОМУ ключу устройства**, а не по транзиентному deviceId: VID+PID(+serial).
   `notifyUvcConnected` — если устройство с таким ключом уже есть, ЗАМЕНИТЬ (обновить id/объект), не
   добавлять второе.
2. Либо: в `notifyUvcConnected` дедупить список по имени/ключу (`distinctBy`); в
   `notifyUvcDisconnected` чистить по тому же ключу.
3. Гарантировать снятие старой записи на detach (смежно bug 35 — detach-путь; если detach не доходит,
   дедуп по ключу всё равно не даст клонов).
4. Приёмка: несколько вкл/выкл Pico+ → в списке ВСЕГДА одна «EMEET Piko+».

## Фикс (2026-07-19)

`DeviceManager.notifyUvcConnected` теперь ДЕДУПит по стабильному ключу устройства **VID+PID** (а не по
транзиентному deviceId): если запись того же устройства уже есть — ЗАМЕНЯЕТ (обновляет id/имя), иначе
добавляет. Гарантия: клонов не будет НИКОГДА, даже если detach-событие не дошло (смежно bug 35). Лог
`UVC dedup: заменил прежнюю запись того же устройства`. Два одинаковых модели-вебки (один VID+PID)
схлопнутся в одну — приемлемо (у нас одна вебка; serial добавим, если понадобится различать).

## Приёмка
- ✅ **Физическая приёмка ПРОЙДЕНА (Криник, 2026-07-19):** после реплага Pico+ «в списке не плодится» —
  клонов больше нет (подтверждено на живом устройстве).
