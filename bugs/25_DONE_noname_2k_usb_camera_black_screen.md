# Bug 25 — Чёрный экран вместо видео с noname «2K USB Camera»

## ✅ DONE (2026-07-07, офис, live-верификация на самом «2K-лимоне»)

**Чёрный экран УСТРАНЁН.** На живой камере (vid=11231 pid=641) приложение показывает стабильное живое
видео 640×360 (композитор всегда рисует базу, тихого отката на встроенную НЕТ — проверено рекурсией
«вебка смотрит на экран»). Крашей/чёрного нет. Скрин-верификация 11:27.

**Root cause «мыла» = ФИЗИКА КАМЕРЫ, не наш баг.** Три контролируемых прогона (правило 3 попыток)
доказали на уровне USB-протокола: камера ЗАЯВЛЯЕТ в дескрипторе до 2560×1440, но реально стримит
только **640×360 MJPEG** — 1080 отклоняется `UVC_SET_CUR err=-9`, запрос 720 резолвится вниз до 640×360.
Полоса/прошивка «лимона». Программно больше не вытащить (MJPEG уже используется, reopen нечего поднимать,
`getCameraRequest` для guard'а непригоден). Решающая форензика — ниже.

**Решение Криника (2026-07-07):** закрыть как есть. 640×360 = потолок ЭТОЙ камеры; чистый 1080 даёт
нормальная вебка (Emeet Piko+). Остаток «не падать молча в fallback при провале негоциации» вынесен в
**idea 20** (выбор источника) как UX-задача. Код оставлен baseline (пробные reopen-правки откачены).

**Статус:** ✅ DONE — чёрный экран решён; «мыло» = аппаратный потолок лимона (не баг); UX-хвост → idea 20.

## Форензика (снята 2026-07-06, logcat старта приложения с воткнутой 2K-камерой)

VID=11231 (0x2BCF) PID=641 (0x0281), class 239/2/1 (UVC), `mHasVideoCapture=true`.
USB attach → permission granted → `UvcCameraOpener.open(1920x1080)` → **нативная негоциация падает:**

```
E/UVCCamera: [UVCCamera.cpp:172:connect]: could not open camera:err=-99
E/UsbDeviceRepository: open failed:result=-99 ... venderId=11231;productId=641
E/UsbDeviceRepository: Camera error: unsupported preview size
I/libUVCCamera: [UVCPreview.cpp:285]: PIXEL_FORMAT_YUV20SP
E/libUVCCamera: [UVCPreview.cpp:524:prepare_preview]: could not negotiate with camera:err=-9
I/UsbDeviceRepository: USB disconnected → Camera closed
```

## Root cause (ПОДТВЕРЖДЁН — гипотеза 1)

Наш `UvcCameraOpener` жёстко просит превью **1920×1080** (`CameraRequest.setPreviewWidth/Height`).
Noname «2K USB Camera» этот размер в YUV НЕ поддерживает → `err=-99 unsupported preview size` →
`could not negotiate err=-9` → камера закрывается → **чёрный экран**. «То чёрный, то работает» =
негоциация иногда проходит на повторной попытке AUSBC (в Phase 3-прогонах 2026-07-06 камера в итоге
ожила и дошла до композитора — см. plans/02 §5).

Опасный побочный эффект: при закрытии камеры в этом состоянии ловили нативный SIGABRT в
`UVCCamera.stopPreview/close/destroy` (crash-буфер 2026-07-05 23:44) — процесс умирал.

## Уточнение форензики (2026-07-06, ночь — попытка фикса через getAllPreviewSizes)

Проверено на живой камере: **`camera.getAllPreviewSizes(null)` ДО открытия возвращает ПУСТО** —
дескрипторы UVC ещё не прочитаны (chicken-and-egg), пред-запрос размера бесполезен (откатил).

Но лог вскрыл главное — **камера НЕ висит на чёрном экране, AUSBC сам восстанавливается:**
```
E/UVCCamera: could not open camera:err=-99          ← наш запрос 1920×1080 YUV не поддержан
E/UsbDeviceRepository: unsupported preview size
I/UsbDeviceRepository: Camera opened
D/UsbViewModel: PreviewStarted(width=640, height=360)  ← AUSBC упал на ближайший YUV-размер
```
Т.е. YUV-превью этой «2K»-камеры максимум **640×360** («2K» у неё только в MJPEG). «Чёрный экран»,
что видел Криник = окно неудачной негоциации ДО фолбэка (или сессия, где фолбэк не успел). После
фолбэка камера доходит до композитора (в Phase 3-прогонах писала валидный композит).

## Дополнение (2026-07-06, офис) — ЗАДАЧА 2 порядка работ

Диагноз уточнён: когда 2K USB валит негоциацию (`result=-99`), приложение НЕ показывает ошибку и НЕ

## Решающая форензика (2026-07-07, офис, живой «2K-лимон» воткнут) — ПОТОЛОК ПОДТВЕРЖДЁН НА ПРОТОКОЛЕ

Снято на живой камере (vid=11231 pid=641), три контролируемых прогона (правило 3 попыток):

**Реальный список размеров (getAllPreviewSizes ПОСЛЕ open):**
`640×360, 1280×720, 1280×960, 1920×1080, 2048×1536, 2560×1440, 640×480`.
Т.е. дескриптор ЗАЯВЛЯЕТ вплоть до 2560×1440. Но по факту:

1. **Первый `open(1920×1080)`** → негоциирует `frameSize=(640,360)@MJPEG` (уже MJPEG!). Причина: на
   первом open дескрипторы ещё не прочитаны → AUSBC `getSuitableSize` не с чем матчить → безопасный
   дефолт 640×360.
2. **Reopen БЕЗ close на 1920×1080** → `could not open camera: err=-99` (устройство занято) → 640×360.
3. **Корректный `close`→пауза→`open(1920×1080)`** → `uvc_probe_stream_ctrl(UVC_SET_CUR): err=-9` →
   `could not negotiate` → откат 640×360. **1080 отклоняется на уровне USB stream-control.**
4. **`open(1280×720)`** → резолвится ВНИЗ до 640×360 (без ошибки, но и без 720).

**Вывод:** практический потолок этой камеры на данном USB — **640×360 MJPEG**. Всё выше заявлено в
дескрипторе, но не стримится (лимит полосы/прошивки — классический «лимон»). Программно из неё больше
НЕ вытащить: reopen-логика не поможет (нет чем поднять), MJPEG уже используется. `getCameraRequest()`
для guard'а бесполезен — возвращает ЗАПРОШЕННЫЙ размер (1920), а не негоциированный (640×360);
негоциированный есть только в событии `UsbViewModel.PreviewStarted`.

**Статус кода:** оставлен baseline (Фаза-2 reopen из коммита) — он корректно помогает камерам, чей
РЕАЛЬНЫЙ максимум < запрошенного, и no-op'ит лимон (что и есть верный исход). Пробные правки откачены.

**Что осталось от bug 25 = НЕ «чёрный экран» (камера доходит до стабильного 640×360, композитор всегда
рисует базу), а два UX-хвоста:** (а) 640×360 апскейлится в 1080-холст = мыльно — это физика лимона, не
софт; (б) «не падать молча в fallback на встроенную». Пункт (б) ниже. Проверить на НОРМАЛЬНОЙ вебке
(Emeet Piko+) — там 1080 стримится чисто, весь пайплайн ок.

### План задачи 2 (после raw-фикса камеры)
- [ ] **Выбор поддерживаемого размера/формата для UVC.** По аналогии с device-камерой: после open
      прочитать `getAllPreviewSizes` (непусто ПОСЛЕ открытия) и, если negotiated << желаемого,
      переоткрыть на лучшем; или включить MJPEG-режим AUSBC (2K отдаёт 2K/1080 в MJPEG). Осторожно с
      нативным close-крашем (bug 28) — минимизировать переоткрытия.
- [ ] **Не падать молча в fallback:** если активный источник — UVC, и он provfailed, не подменять
      его тихо встроенной камерой. Варианты: показать состояние «камера не открылась» (как standby-
      слой) ИЛИ явно переключить с уведомлением. UX — согласовать (это ближе к idea 20 выбор источника).
- [ ] Проверить на нормальной MJPEG-вебке (не 2K-лимоне) — что источник работает чисто.
