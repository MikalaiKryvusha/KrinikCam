## Архитектурное решение (backend компоновки) — выбрано агентом

Свой GL-композитор НЕ пишем. Backend = штатная система фильтров RootEncoder
(`GlStreamInterface.addFilter/removeFilter/clearFilters` = z-order + видимость;
`BaseObjectFilterRender.setScale/Position/Alpha/Rotation` = трансформа для будущих фаз). Готовые
рендереры покрывают почти все типы Q2: `ImageObjectFilterRender` (PNG), `TextObjectFilterRender`,
`GifObjectFilterRender`, `SurfaceFilterRender` (видео/анимации/2-я камера), `AndroidViewFilterRender`
(браузер). Камера — базовый VideoSource (низ), оверлеи — фильтры поверх (90% кейсов). Доменная
модель (Layer/Scene/SceneProfile) и UI стабильны и не зависят от backend — если позже понадобится
произвольный z-order/композит нескольких камер, backend можно заменить, не трогая модель/UI.
Обоснование: KISS (PHILOSOPHY.md) — не изобретаем то, что библиотека уже умеет.

## Контекст / что уже выяснил по коду

- Сейчас в пайплайн RootEncoder подаётся **ОДИН** `VideoSource`, рисующий в одну GL-поверхность
  энкодера. Источники взаимоисключающие и переключаются через `changeVideoSource`:
  `UvcCamera` (USB), `Virtual` (тест-паттерн), `Standby`/`FreezeStandby` (заглушка), `None`.
  Файлы: `feature/capture/.../model/VideoSource.kt`, `feature/streaming/.../rtmp/*`.
- Чтобы получить OBS-слои, нужен **наш собственный GL-композитор**: один «составной» источник,
  который владеет СПИСКОМ слоёв и каждый кадр компонует их (z-order, видимость, трансформ) в
  поверхность энкодера. Это новый фундаментальный модуль — на нём потом строится всё (в т.ч.
  «Please stand by» как частный случай слоя-fallback).
- Технически реализуемо нашими силами (GL мы уже держим — VirtualFrameRenderer/StandbyFrameRenderer
  рисуют в Surface). Главный вопрос не «можно ли», а **насколько широко берём в первый заход** и
  **как стример этим управляет** — это UX/архитектура, решает Криник.

## ВОПРОСЫ

### Q1. Насколько широкий фундамент берём ПЕРВЫМ заходом?
- **A) (рекомендую)** Минимальный вертикальный срез: GL-композитор + модель слоёв (список, z-order,
  видимость) на 2 типах источников — **камера** + **картинка/оверлей (PNG)**. Доказываем пайплайн
  компоновки целиком, дальше типы добавляются дёшево. Слои пока **во весь кадр** (трансформа нет).
- **B)** Сразу OBS-lite: камера + картинка + текст + цвет-фон + 2-я камера, со свободным
  перемещением/масштабом на канвасе. Большой объём, риск надолго застрять.
- **C)** Сначала только перенести «Please stand by» в модель источника как fallback-слой, без общей
  системы слоёв. Меньше пользы, но снимает краш-сценарий быстрее.

**Ответ:** А для "доказать пайплайн", а потом расширять будем.

### Q2. Какие типы источников нужны тебе как стримеру в первую очередь? (можно несколько)
- **Камера (USB UVC)** — основной
- **Картинка/оверлей (PNG с прозрачностью)** — логотип, рамка, «сейчас вернусь»
- **Текст** — подписи, ник, бегущая строка
- **Цвет/градиент-фон** — подложка
- **Вторая камера** (фронталка телефона) — PiP «лицо в углу»
- **Захват экрана** устройства

**Ответ:** 
- USB UVC
- Картинка/оверлей (PNG с прозрачностью)
- Fallback заглушка на отключение камеры
- Видео-оверлей
- Текст
- Вторая камера, чаще всего это будет встроенная камера девайса, фронталка или селфи.
- Анимации-реакции, типа сердечко (частный случай видео оверлея с прозрачным фоном видео, как стикеры в Телеграме)
- Браузер (выглядит очень большой фичей, можно сильна на потом оставить)
- Захват экрана - минимум приоритет, на будущее.

### Q3. Как стример управляет рабочей областью (UI/UX)?
- **A) (рекомендую)** Отдельная панель «Слои/Сцена»: список источников (перетаскивание для порядка,
  «глаз» вкл/выкл, тап → настройки слоя), а полноэкранный вьюфайндер показывает результат компоновки.
  Привычно, не загромождает живой кадр, хорошо тапается на планшете.
- **B)** Прямое управление на полноэкранном вьюфайндере: тап по слою → ручки перемещения/масштаба
  прямо на живом канвасе (OBS на тач). Нагляднее, но сложнее и рискованно для случайных тапов в эфире.
- **C)** Оба: панель-список + базовый drag по канвасу.

**Ответ:** А

### Q4. Как сюда вписывается «Please stand by» (заглушка при пропаже камеры)?
- **A) (рекомендую)** Системный fallback-слой: всегда присутствует скрытым, **авто-появляется** когда
  слой-камера теряет сигнал (по правилам таймаута/freeze из interview_004), **авто-прячется** при
  возврате. Пользователь не может его удалить, но может заменить картинку/настроить таймаут.
- **B)** Обычный пользовательский слой: стример сам добавляет «заглушку» и сам решает её правила.
  Гибко, но он может забыть добавить → в эфире «дыра» при отвале камеры.

**Ответ:** Особый источник медиа "ЗАГЛУШКА KrinikCam", который по-умолчанию всегда есть в новом "ПРОФИЛЕ СЦЕНЫ" с дефолными настройками - по желанию его можно удалить из профиля сцены, тогда при отключении камеры будет просто чёрный экран - не будет работать логика этого особого источника мидеа ЗАГЛУШКА KrinikCam.
Процили сцены тоже делаем, тоже их можно импортировать и экспортировать в/из файла - у нас по всему приложению будут профили для разных сущностей.

## Предлагаемый план реализации (после ответов)

1. Новый модуль/класс **`CompositorVideoSource`** — единый VideoSource для энкодера, владеет
   `List<Layer>`; GL-цикл компонует слои по z-order/visibility/transform в Surface энкодера.
2. Модель **`Layer`** (sealed): тип источника + общие свойства (id, name, zIndex, visible,
   transform). Рендереры по типу (камера-текстура, bitmap, текст, цвет).
3. Интеграция в `RtmpStreamer`: вместо свопа источников — один `CompositorVideoSource`; старые
   Virtual/Standby/Freeze становятся слоями.
4. UI по выбранному в Q3 варианту (панель «Слои» и/или канвас-манипуляция).
5. «Please stand by» переносится в fallback-слой (Q4) → попутно закрывает гонку детача из Bug 13.
6. Тесты на устройстве через ADB-харнесс (вирт. камера, разрыв источника) без физической камеры.
