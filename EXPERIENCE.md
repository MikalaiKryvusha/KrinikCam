# EXPERIENCE — накопленный опыт агента

> Растущий журнал уроков агента. **Вынесенная память о том, *что работает, а что нет*** — чтобы свежая
> сессия без контекста (или автономный цикл) не наступала повторно на те же грабли. Сверяйся ДО задачи;
> дописывай ПОСЛЕ значимой попытки (успех **или** провал). Ищи грепом, не листай.
>
> **Теги живут инлайн на каждой записи** (центрального списка тегов нет) — один греп находит опыты
> напрямую: `grep '#build' EXPERIENCE.md` · `grep -i '#usb\|#uvc' EXPERIENCE.md` ·
> `grep '❌' -A4 EXPERIENCE.md` · `grep 'EXP-0003' EXPERIENCE.md`. Переиспользуй существующий тег,
> если подходит (грепни файл, чтобы увидеть, какие в ходу).
>
> **Формат записи (коротко и греп-дружелюбно).** Новые сверху. Каждая запись начинается со стабильного
> id, ISO-даты, маркера итога (`✅` / `❌` / `❌→✅`) и инлайн-`#тегов`:
>
> ```
> ### EXP-0001 · 2026-01-01 · ✅ · #тег #область
> **Контекст:** одна строка — что делалось.
> **Пробовал / сделал:** подход, кратко.
> **Результат:** ✅/❌ — что вышло.
> **Урок:** переиспользуемый вывод (причина существования записи).   → см.: bugs/NN · ideas/NN · plans/NN
> ```
>
> Навык: `/experience` (зафиксировать урок · вспомнить релевантные уроки).
> Граница с `bugs/`: там — документ на дефект (симптом→форензика→фикс); здесь — короткие кросс-задачные
> уроки уровня ПОДХОДА (включая успехи). Живой справочник — тегом DONE не помечается.

Собирай маску типов динамически от фактического состояния устройства. И: краш такого рода ловится ТОЛЬКО живой приёмкой — билд зелёный, ревью молчит.   → см.: bugs/36, plans/10

### EXP-0012 · 2026-07-12 · ✅ · #streaming #forensics #network #logcat
**Контекст:** «краш после долгого эфира (~1 мин)→стоп» (хвост bug 34); ловил его на build 19 (со фиксом S1) двумя долгими прогонами на RTMP-полигоне с непрерывным logcat в файл.
**Пробовал / сделал:** не гадал «где падает», а собрал форензику по таймлайну logcat: битрейт `onNewBitrate` ушёл в 0; и видео, и аудио RTMP-пакеты (`RtmpSender: wrote X packet`) встали ОДНОВРЕМЕННО (единый sender-тред завис на `write()`); превью-поверхность (`BufferQueueProducer …MainActivity#…`) при этом держала ~28 fps; сервер закрыл соединение по `read i/o timeout`, приложение получило `Broken pipe`.
**Результат:** ✅ — краш НЕ воспроизвёлся (S1 корректно восстанавливает failed-путь), а сам обрыв опознан как СЕТЕВОЙ БЛИП: 28 fps превью доказали, что GL/энкодер/камера не при чём — стопор НИЖЕ энкодера, на сокете.
**Урок:** «стрим умер» диагностируй по таймлайну, а не по интуиции. Ключевые сигналы: bitrate→0 + одновременная остановка audio+video sender = сеть/сокет (не энкодер); превью-fps в logcat (`BufferQueueProducer`) — независимый пульс GL-пайплайна, отделяет «энкодер голодает» от «сеть встала». Для таких приёмок — непрерывный `adb logcat -v time` в файл ДО прогона, не `logcat -c` после.   → см.: bugs/34, plans/09
