## Цель эпика

Сцена (набор слоёв с источниками / трансформами / z-order / видимостью / оверлеями) **сохраняется,
переживает рестарт, экспортируется/импортируется**; можно держать НЕСКОЛЬКО сцен и **бесшовно
переключаться** между ними, в т.ч. в эфире — из отдельного FAB-радиалки. Это фундамент «мобильного
OBS» ([[idea 25]]): без сохранения и переключения сцен многослойная работа теряется при каждом рестарте.

## Что уже есть (опора)

- `Scene(layers: List<Layer>)` иммутабельна; `Layer.VideoCapture(source: CaptureSource, transform, visible)`,
  `Layer.Image(bitmap, transform, visible)`; `LayerTransform(scale/cx/cy/alpha/rotation)`. Доккоммент
  `Scene.kt` уже закладывал «сериализацию в профиль сцены».
- **Стример = единый источник правды сцены:** `RtmpStreamer._scene: MutableStateFlow<Scene>`
  (`@Singleton`, есть `scope` с веткой `Dispatchers.IO`). Все правки идут через `mutateScene {…}` →
  `applySceneLayers()`. Сюда вешаем автосейв (на каждую правку) и restore (на старте).
- **Live-смена слоёв:** `CompositorVideoSource.setLayers` (через `applySceneLayers`) — переключение
  сцены без рестарта энкодера ложится сюда.
- **Паттерн персиста + экспорт/импорт:** `:data:profiles` (Room + DataStore, `buildExportJson` /
  `importProfilesFromJson`). **`:feature:streaming` уже зависит от `:data:profiles`** → персист сцены
  кладём туда, стример читает напрямую.
- **Паттерн одиночного JSON-конфига в DataStore:** `DeviceProfile` хранится как JSON-строка
  (`ProfilesDataStore`). Для «одной текущей сцены» (Фаза 0) — тот же приём, БЕЗ Room-миграции.
- **Паттерн UI-радиалки:** `ui/overlay/FloatingRadialMenu.kt`. Паттерн менеджера:
  `StreamPlatformsOverlay` / `EncoderProfilesOverlay`. FAB слоёв — внизу-слева (`MainScreen`,
  `Alignment.BottomStart`), рядом ставим FAB сцен.

## Сквозные технические решения (общие для фаз)

1. **Хранилище растёт по фазам:** Ф0 — JSON-снапшот ОДНОЙ текущей сцены в DataStore (как `DeviceProfile`,
   миграции не нужны). Ф1 — Room-таблица `scene_profiles` (набор именованных сцен) + активный id в
   DataStore (как `active_profile_id`).
2. **Оверлеи-картинки → файлы.** `Layer.Image` держит `Bitmap` в памяти — в снапшоте храним ПУТЬ
   (`filesDir/overlays/<layerId>.png`), а не пиксели. Файл пишется ОДИН раз (слой-картинка иммутабельна),
   на восстановлении грузится из файла; сироты чистятся.
3. **Дуализм источника дефолтного слоя `camera`.** Источник слоя уже живёт в модели
   (`Layer.VideoCapture.source`), а восстановление опенеров идёт через существующий `LaunchedEffect` в
   `MainScreen` (ключ `extraCameraLayers.map { id to source }`). Условие полного восстановления: выбранный
   источник (в т.ч. АВТО-выбор на старте) должен ПОПАСТЬ в `scene.source` (через `setCameraLayerSource`).
   Проверить/довести до Ф0 — иначе restore камеры неполный.
4. **Модуль DTO.** Сериализуемый снапшот (`SceneSnapshotDto` и слои) ссылается на доменные типы сцены →
   живёт в `:feature:streaming` (`scene/persist/`). `:data:profiles` хранит его как ОПАКОВУЮ JSON-строку
   (не знает про типы сцены — слои зависимостей не разворачиваются).
5. **Переключение сцены в эфире** (Ф2) — зона GL/камер-lifecycle (семья bug 58/62/63): камеры
   переоткрывать ТОЛЬКО при реальной смене физ-источника (`sourceKey`), энкодер не рвать.

### Фаза 0 — MVP: сцена переживает рестарт + FAB-якорь
Определение готово Криником: **«UI добавляем FAB и научаемся сохранять настроенную сцену между сессиями
запуска приложения».** Разрезает старые Ф0+Ф1 (сериализация + персист) до ОДНОЙ текущей сцены и ставит
FAB-якорь из Ф2 (минимальный). Проверяет самый рискованный кусок (сериализация + оверлей→файл + restore +
дуализм источника) сквозняком. Детали → [18_phase0_mvp.md](18_phase0_mvp.md).
**Приёмка:** собрать сцену 3+ слоёв (камера + 2 оверлея, разные трансформы) → `am force-stop` → запустить →
сцена восстановлена 1:1 (дамп + скриншот); FAB виден и работает; unit round-trip зелёный.

### Фаза 1 — Набор именованных сцен + менеджер
Room-таблица `scene_profiles` (CRUD) + активный id в DataStore. FAB-радиалка становится настоящим
менеджером: веер сцен (тап = переключить) + «Новая / Дублировать / Переименовать / Удалить». Автосейв Ф0
привязывается к АКТИВНОЙ записи. Детали → [18_phase1_scene_set.md](18_phase1_scene_set.md).

### Фаза 2 — Бесшовное переключение в эфире
Переключение активной сцены во время стрима: `setLayers` уже live; камеры переоткрывать ТОЛЬКО при
реальной смене физ-источника (`sourceKey`, bug 58). Не рвать энкодер, без чёрного/мельтешения (учесть
bug 62/63). Детали → [18_phase2_live_switch.md](18_phase2_live_switch.md).

### Фаза 3 — Экспорт / импорт набора сцен
Экспорт: JSON (как платформы) + картинки (zip или встроенные ассеты — по развилке 2 [[idea 40]]). Импорт:
слить/добавить сцены без затирания (как `importProfilesFromJson`). Детали →
[18_phase3_export_import.md](18_phase3_export_import.md).

## Соответствие старой разбивке (было → стало)

Старый план 18 разбивал эпик на Ф0-Ф4 «по слоям» (сначала вся модель, потом весь персист, потом весь UI).
Криник перекроил «по ценности» (каждая фаза — рабочий срез end-to-end):

- Старые **Ф0 (доменная модель+сериализация)** + **Ф1 (персист автосейв одной сцены)** + якорь из **Ф2** →
  новая **Фаза 0** (для ОДНОЙ текущей сцены).
- Старая **Ф2 (FAB + радиалка)** в части менеджера набора → новая **Фаза 1**.
- Старая **Ф3 (переключение в эфире)** → новая **Фаза 2**.
- Старая **Ф4 (экспорт/импорт)** → новая **Фаза 3**.

## Риски / внимание (сквозные)

- Оверлеи как bitmap-в-памяти → перевести на файлы (реш. 2), иначе сцена не сериализуется.
- Дуализм источника дефолтного слоя `camera` (activeSource vs scene.source, реш. 3) — развести в Ф0,
  иначе восстановление камеры неполное.
- **Автосейв на КАЖДУЮ правку сцены** — `mutateScene` зовётся ЧАСТО (каждый кадр жеста трансформы) →
  обязателен debounce, иначе спам записи в DataStore/файлы и просадка жестов.
- Переключение сцен в эфире (Ф2) — зона GL/камер-lifecycle (семья bug 58/62/63): аккуратно, с
  харнес-стрессом.
- Секретов в сцене нет (ключи — в профилях платформ), но при экспорте с ассетами (Ф3) следить за
  размером/приватностью.

## Открытые развилки Кринику (из [[idea 40]] — какую фазу блокируют)

1. **Автосейв vs явное сохранение** — в Ф0 закрыто de-facto: одна «текущая» сцена всегда автосейвится
   (это и есть определение Ф0). Именованные профили (Ф1) — отдельные записи.
2. **Экспорт: одна сцена или весь набор? zip с картинками сейчас или позже?** — блокирует **Фазу 3**
   (интервью перед стартом Ф3).
3. **Радиалка при МНОГИХ сценах: сколько показывать веером, нужен ли отдельный полный список-менеджер?**
   — блокирует **Фазу 1** (интервью перед стартом Ф1).
