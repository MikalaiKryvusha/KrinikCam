# Idea 06 — Поворот видео + смена соотношения сторон (16:9 ↔ 9:16)

> Статус: **✅ ЗАКРЫТА** (2026-06-29). Превью-поворот ✅ (все комбинации девайс×камера), ландшафтный
> 16:9 стрим ✅, **портретный 9:16 стрим ✅ — подтверждено ЖИВЬЁМ на YouTube** (Bug 10 решён). Горячие
> повороты девайса в эфире не рвут стрим. Полная история фикса портрета → `bugs/10_*` (раздел
> «🏆 ЭПОПЕЯ»). Решение: `RtmpStreamer.configureCaptureRotation` — портретный канвас + `setIsPortrait`
> + поворот в источнике (`RotatableSource`) или `setCameraOrientation` для камеры; энкодер rotation=0.
>
> Реализует «горячую кнопку поворота» из STATUS-фидбэка, но с расширенной семантикой: поворот
> меняет не только картинку, но и **формат потока**.

## Итоговая архитектура (важно для будущих сессий)

Превью и стрим крутятся **раздельно**, иначе двойной поворот / искажение:
- **Превью (то, что на экране)** — display-only поворот через **матрицу на TextureView**
  (`UvcPreviewView.applyPreviewRotation`): GL всегда рисует ровный ЛАНДШАФТ, а вид поворачивается
  и масштабируется (`postRotate` + `postScale(min/max)`) → центрированный портрет 9:16 с чёрными
  полями по бокам, БЕЗ искажения. Это и есть «как делают все камеро-приложения».
- **Стрим (кодируемый выход)** — `glInterface.setStreamRotation(deg)` (только энкодер, НЕ превью)
  + swap размеров энкодера в `prepareVideo` (1920×1080 → 1080×1920 при 90/270).
- Вход камеры держим ровным: `setCameraOrientation(0)` (Bug 02 A).
- **Тупик, которого избегать:** НЕ крутить превью через `setCameraOrientation`/portrait-энкодер в
  GL — это давало растяжение (encoder portrait на landscape-поверхности), а Compose
  `aspectRatio`/смена размера энкодера на лету → чёрный экран. Матрица на TextureView — верный путь.

Файлы: `RtmpStreamer` (setVideoRotation/applyVideoRotation/rotatedDims/prepareVideo),
`UvcPreviewView` (applyPreviewRotation + rotationDegrees), `RotationMenu.kt` (UI),
`MainScreen` (videoRotation → RotationMenu + UvcPreviewView), `StreamingRepository`/`StreamViewModel`.

## Суть фичи

Кнопка поворота на главном экране (вверху справа) раскрывается в **радиальное мини-меню**
(по аналогии с FAB-меню) с выбором угла: **0° / 90° / 180° / 270°**.

Поворот меняет **соотношение сторон выходного видео**, а не просто крутит картинку в той же рамке:
- **0° / 180°** → ландшафт **16:9** (1920×1080)
- **90° / 270°** → портрет **9:16** (1080×1920)

Технически: мало повернуть вход камеры (`glInterface.setCameraOrientation(deg)` — это даст
повёрнутую картинку с леттербоксом внутри ландшафтной рамки). Нужно **менять размеры энкодера
местами** (swap width↔height), чтобы наружу уходил настоящий портретный/ландшафтный кадр.

## Зачем (приоритет использования)

1. **Будущие вертикальные трансляции** — Reels / Shorts / TikTok-формат 9:16.
2. **Локальная запись с камеры в файл** на устройство (будущая фича) — в нужной ориентации.
3. Сейчас — базовая возможность повернуть ландшафтную USB-вебку под нужный формат до эфира.

## Решения Криника (интервью в чате, 2026-06-28)

| Вопрос | Решение |
|--------|---------|
| UI кнопки поворота | **Радиальное мини-меню** с выбором угла (0/90/180/270), как у FAB Menu |
| Что меняет поворот | **Соотношение сторон потока** (16:9 ↔ 9:16), а не только картинку |
| Когда меняем формат | В **превью** и при **старте стрима** (`prepareVideo` с обменянными размерами) |
| **Поворот во время эфира** | **ЗАПРЕЩЁН.** Кнопка неактивна, пока идёт трансляция. Хочешь повернуть — **стоп стрим → поворот → старт заново.** |
| Изучить реакцию YouTube на смену формата на лету | Изучено сейчас (см. ниже) — подтверждает запрет смены в эфире |

## Research: смена разрешения/соотношения RTMP на лету (2026-06-28)

Вывод: **менять разрешение/соотношение на активном RTMP-соединении — рискованно и не
поддерживается как штатная практика.**
- YouTube определяет параметры энкодера (разрешение, fps) **на старте** трансляции; рекомендован
  постоянный CBR и фиксированное разрешение на весь эфир.
- «Changing the source encoding configuration while using a same opened channel can cause issues» —
  т.е. смена конфигурации энкодера на том же открытом канале может оборвать/испортить поток.
- Для смены формата корректный путь — остановить и пересоздать трансляцию.

→ Полностью подтверждает решение «во время эфира поворот блокируем».

## План реализации

1. **`RtmpStreamer`** — хранимый угол `videoRotation` (StateFlow), `setVideoRotation(deg)`:
   - swap размеров энкодера для превью (`setEncoderSize`) + рестарт превью для реинициализации GL;
   - `prepareVideo` при старте стрима получает обменянные размеры при 90/270;
   - `applyVideoRotation()` (= `setCameraOrientation(deg)`) переживает рестарты превью/стрима/standby;
   - **блок смены угла, пока `isStreaming` == true** (no-op + лог/snackbar).
2. **`StreamingRepository` / `StreamViewModel`** — проброс `videoRotation` + `setVideoRotation`.
3. **UI** — раскрывающееся радиальное меню углов на главном экране; **неактивно во время эфира**.
4. Тест на устройстве: превью 16:9 → выбор 90° → превью 9:16 портрет; Go Live → поток портретный.

## Портретный стрим: провал и зацепки (2026-06-29) — ОТКРЫТО

Превью 9:16 работает (матрица TextureView), но КОДИРУЕМЫЙ выход 9:16 — нет. Три живых теста на
YouTube, у каждого свой режим искажения (камера = USB UVC 16:9 landscape через кастомный
`UvcVideoSource`):

| Попытка | Конфиг энкодера | Результат на YouTube + превью |
|---------|-----------------|-------------------------------|
| 1 | размер 1080×1920 (swap) + `setStreamRotation(90)` | сжато в центр (контент не повёрнут, вписан в портрет) |
| 2 | размер 1920×1080 (без swap) + `setStreamRotation(90)` | повёрнуто, но размеры остались 16:9 → YouTube растягивает по ширине |
| 3 | размер 1080×1920 (swap) + `setCameraOrientation(90)` | портрет, но **сжат по вертикали** (и на устройстве, и на YouTube) |

**Ключевой инсайт:** при попытке 3 источник, похоже, сначала ВПИСЫВАЕТСЯ в портретный канвас
(сжимается по горизонтали → выглядит как вертикальное сжатие после поворота), и только потом
поворачивается. То есть `AspectRatioMode.Adjust` НЕ применяется к энкодеру на повёрнутом пути
(работает только для превью). Также YouTube Studio МЕДЛЕННО авто-детектит разрешение и «гуляет»
между видами → итерации вслепую ненадёжны.

**Зацепки для следующей (сфокусированной) сессии:**
1. **Не пробованный штатный путь:** `prepareVideo(w, h, bitrate, fps, iFrame, rotation=90)` — 6-й
   параметр `rotation` это ОФИЦИАЛЬНЫЙ способ портрета в RootEncoder (issue #407: юзер звал
   `prepareVideo(1280,720,...,90)`). Передавать ИСХОДНЫЕ landscape-размеры + rotation=90, НЕ свапать
   руками, НЕ звать setStreamRotation/setCameraOrientation. ОСТОРОЖНО: prepareVideo внутри зовёт
   `setCameraOrientation` по логике телефонного сенсора (Bug 02 A: при rotation=0 ставит 270, мы
   форсим 0). Для USB-камеры (upright) будет offset — подобрать коррекцию ПОД rotation=90.
2. Изучить ИСХОДНИКИ RootEncoder 2.4.7 (`GlStreamInterface`, `StreamBase.prepareVideo`,
   `RotationFilterRender`) — как именно aspect/rotation применяются к ЭНКОДЕРУ (не превью) при
   кастомном VideoSource. AAR распакован ранее: `scratchpad/classes.jar` (javap).
3. Тестировать БЕЗ YouTube-flip-flop: записывать локально или ffprobe RTMP, чтобы видеть истинный
   выход энкодера, а не лагающий Studio-монитор.

Текущее состояние кода (зафиксировано): попытка 3 (portrait swap + setCameraOrientation в эфире).
Ландшафтный стрим (0°) при этом РАБОТАЕТ корректно — ничего не сломано для основного сценария.
