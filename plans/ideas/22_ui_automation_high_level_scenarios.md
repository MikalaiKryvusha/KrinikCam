# Idea 22 — UI-автоматизатор: высокоуровневые сценарии (быстро, умно, надёжно)

> Источник: Криник, чат 2026-06-29. Возникло по ходу Idea 21: ИИ-агент тратит много шагов и «думает»
> над каждым тапом (открыть Settings → проскроллить → лонг-тап → найти тумблер…), а тапы по тумблерам
> иногда не срабатывают (ненадёжно). Нужен инструмент уровня НАМЕРЕНИЯ, а не отдельных тапов.
> Статус: 🟡 запланировано (инструментальная инфраструктура для автономной разработки/тестов).

## Проблема

Сейчас `tools/ui.mjs` — низкоуровневый: `dump`/`tap <text>`/`swipe`/`longpress`. Чтобы, например,
включить виртуальную камеру, агент вручную оркеструет цепочку: FAB→Settings→scroll→longpress
«KrinikCam»→Developer→tap «Virtual camera». Это:
- медленно (много раунд-трипов агент↔устройство),
- хрупко (тап по тумблеру не всегда срабатывает; скролл может не докрутить; состояние экрана разное),
- заставляет агента «думать» над каждым шагом вместо сценария.

## Чего хотим

Команды уровня **намерения/сценария**: агент говорит ОДНОЙ командой «хочу X», а инструмент сам,
ИЗ ЛЮБОГО текущего состояния, находит путь к нужному элементу и достигает цель — быстро, умно, надёжно.

Примеры целевых команд:
```
node tools/ui.mjs scenario virtual-camera on|off|toggle
node tools/ui.mjs scenario stream-to-file on|off
node tools/ui.mjs scenario go-live            # запустить стрим/запись
node tools/ui.mjs scenario stop               # остановить
node tools/ui.mjs scenario add-test-overlay
node tools/ui.mjs scenario open-layers
node tools/ui.mjs scenario set-rotation 90
node tools/ui.mjs scenario goto settings|developer|main|layers
```
Каждый сценарий: знает целевой экран и элемент, сам туда НАВИГИРУЕТ (открыть нужный экран, проскроллить
до элемента, при необходимости вернуться назад), ПРОВЕРЯЕТ результат (тумблер реально переключился,
экран открылся) и РЕТРАИТ при неудаче. Возвращает чёткий статус (done/already/failed + почему).

## Принципы реализации (как сделать надёжным)

1. **Навигация по состоянию, а не по координатам.** Из `dump` определять «где я сейчас» (по якорным
   текстам каждого экрана) → строить маршрут к цели (граф экранов: main↔settings↔developer, оверлеи).
2. **Идемпотентность тумблеров.** Перед тапом читать текущее состояние тумблера (checked/desc), тапать
   ТОЛЬКО если нужно; после тапа ПЕРЕЧИТАТЬ и убедиться, что переключилось; иначе ретрай (N раз).
   Решает «тап не сработал».
3. **Надёжный тап по тумблеру.** Тапать по центру строки/самого Switch (а не только по тексту); если
   Switch имеет свой bounds — бить по нему. Возможно поднять через `dump` точные bounds Switch-виджета.
4. **Скролл-до-элемента.** Если элемент не виден — свайпить в нужную сторону до появления (с лимитом
   попыток), затем действовать.
5. **Якоря экранов.** Таблица: экран → характерные тексты (для распознавания) + как на него попасть
   (кнопка/жест) + как уйти. Это ядро «графа навигации».
6. **Скорость.** Минимум скриншотов (использовать `dump`, он быстрый); батчить проверки; не спать
   лишнего (поллить состояние короткими интервалами вместо фиксированных sleep).
7. **Чёткий результат в stdout** — агенту достаточно одной строки, без анализа дампов.

## Что это даёт

- Агент тестирует сценарии ОДНОЙ командой → быстрее автономные циклы (особенно Idea 21: «включи
  виртуалку», «запиши в файл», «поставь поворот 90», «добавь оверлей»).
- Меньше «сломанного телефона» агент↔устройство, меньше токенов на оркестрацию тапов.
- Расширяемость: новые сценарии добавляются по мере роста приложения.

   `goto developer|settings|main`, `open-layers`, `add-test-overlay`, `set-rotation`.
3. Дальше — по мере появления функций (мульти-стрим, профили сцены и т.д.).
