# Idea 22 — Надёжное управление приложением для тестов: debug-broadcast-команды (основа) + тонкий UI-автоматор

> Источник: Криник, чат 2026-06-29. Возникло по ходу Idea 21: ИИ-агент тратит много шагов и «думает»
> над каждым тапом (Settings → scroll → long-tap → найти тумблер…), а тапы по тумблерам ненадёжны.
> **Решение (Криник + агент):** основной слой управления состоянием — ADB **debug-broadcast-команды**
> (очень надёжно/быстро/детерминированно); UI-автоматор оставляем ТОНКИМ и зовём только для проверки
> самого UI.
> Статус: 🟡 запланировано (инфраструктура для автономной разработки/тестов).

## Почему два слоя, а не один

Это РАЗНЫЕ задачи:
- **«Загнать приложение в нужное состояние, чтобы протестить пайплайн»** (90% автономных циклов: типа
  «вирт.камера ON + стрим в файл → снять ffprobe») — тут важна НАДЁЖНОСТЬ и СКОРОСТЬ, а сам UI не важен.
- **«Проверить, что работает сам UI»** (тапы/навигация/тумблеры реально переключают) — тут нужен
  реальный пользовательский путь.

ADB-броадкасты идеальны для первого (надёжно, быстро, без навигации). UI-автоматор нужен для второго
(ловит баги именно UI), но по природе медленнее/хрупче. Поэтому: **broadcasts — основа state-control,
UI-автоматор — точечная проверка интерфейса.**

## A. Debug-broadcast-команды (ОСНОВА) — `tools/ui.mjs cmd <name> [args]`

Один **command-receiver** в приложении (debug-only, как уже есть `SET_VIRTUAL_CAM`): принимает
`action + extras`, диспетчеризует в нужный метод VM/репозитория напрямую (минуя UI). Расширяемый список.

Примеры (агент зовёт ОДНОЙ командой, состояние меняется детерминированно):
```
node tools/ui.mjs cmd virtual-camera on|off
node tools/ui.mjs cmd stream-to-file on|off
node tools/ui.mjs cmd go-live              # старт стрима/записи (по тек. настройкам)
node tools/ui.mjs cmd stop
node tools/ui.mjs cmd add-overlay [test|<path>]
node tools/ui.mjs cmd remove-layer <id>
node tools/ui.mjs cmd toggle-layer <id>
node tools/ui.mjs cmd set-rotation 0|90|180|270
```
Под капотом: `adb shell am broadcast -a com.kriniks.kcam.CMD --es action <name> [--es ... ] -p <pkg.debug>`.
Приёмник в `MainActivity`/Application (как текущий `SET_VIRTUAL_CAM`-receiver), вызывает соответствующие
методы. `tools/ui.mjs cmd` — тонкая обёртка, печатает результат.

**Требования:**
- **Debug-gated:** receiver регистрируется/срабатывает только в debug (или гейт по `BuildConfig.DEBUG`) —
  чтобы бэкдор не попал в release.
- **Идемпотентность + ответ:** команда возвращает done/already/failed (через лог или result-extra).
- **Единый receiver-диспетчер**, а не по приёмнику на фичу (текущий `SET_VIRTUAL_CAM` влить сюда).

## B. UI-автоматор (ТОНКИЙ) — только проверка интерфейса

Остаётся низкоуровневым (`dump`/`tap`/`swipe`/`longpress`) + немного надёжных хелперов, и зовётся ТОЛЬКО
когда цель — проверить сам UI (что тумблер/навигация реально работают для пользователя). Не пытаемся
строить из него полный «умный навигатор» — это делают broadcasts.

Минимальные улучшения надёжности (по необходимости): idempotent-toggle (читать состояние Switch до/после
тапа + ретрай), скролл-до-элемента, тап по центру Switch-виджета (а не только по тексту).

## План (итеративно)

1. **Command-receiver (debug)** в приложении: единый `com.kriniks.kcam.CMD` + диспетчер; влить
   `SET_VIRTUAL_CAM`. Методы: virtual-camera, stream-to-file, go-live, stop, set-rotation, add/remove/
   toggle-layer.
2. **`tools/ui.mjs cmd <name>`** — обёртка над `am broadcast`, печать результата.
3. Подключить к текущим нуждам Idea 21 (вирт.камера/запись/поворот/оверлеи одной командой).
4. UI-автоматор — точечные надёжные хелперы по мере необходимости (проверка самих экранов).
5. Обновить `tools/UI_AUTOMATION_GUIDE.md` и таблицу в `AGENT_GUIDE.md`.
