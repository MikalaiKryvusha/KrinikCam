# Idea 06 — Поворот видео + смена соотношения сторон (16:9 ↔ 9:16)

> Статус: **🔧 В РАБОТЕ** (начато 2026-06-28). Реализует «горячую кнопку поворота» из STATUS-фидбэка,
> но с расширенной семантикой: поворот меняет не только картинку, но и **формат потока**.

---

## Суть фичи

Кнопка поворота на главном экране (вверху справа) раскрывается в **радиальное мини-меню**
(по аналогии с FAB-меню) с выбором угла: **0° / 90° / 180° / 270°**.

Поворот меняет **соотношение сторон выходного видео**, а не просто крутит картинку в той же рамке:
- **0° / 180°** → ландшафт **16:9** (1920×1080)
- **90° / 270°** → портрет **9:16** (1080×1920)

Технически: мало повернуть вход камеры (`glInterface.setCameraOrientation(deg)` — это даст
повёрнутую картинку с леттербоксом внутри ландшафтной рамки). Нужно **менять размеры энкодера
местами** (swap width↔height), чтобы наружу уходил настоящий портретный/ландшафтный кадр.

## Зачем (приоритет использования)

1. **Будущие вертикальные трансляции** — Reels / Shorts / TikTok-формат 9:16.
2. **Локальная запись с камеры в файл** на устройство (будущая фича) — в нужной ориентации.
3. Сейчас — базовая возможность повернуть ландшафтную USB-вебку под нужный формат до эфира.

---

## Решения Криника (интервью в чате, 2026-06-28)

| Вопрос | Решение |
|--------|---------|
| UI кнопки поворота | **Радиальное мини-меню** с выбором угла (0/90/180/270), как у FAB Menu |
| Что меняет поворот | **Соотношение сторон потока** (16:9 ↔ 9:16), а не только картинку |
| Когда меняем формат | В **превью** и при **старте стрима** (`prepareVideo` с обменянными размерами) |
| **Поворот во время эфира** | **ЗАПРЕЩЁН.** Кнопка неактивна, пока идёт трансляция. Хочешь повернуть — **стоп стрим → поворот → старт заново.** |
| Изучить реакцию YouTube на смену формата на лету | Изучено сейчас (см. ниже) — подтверждает запрет смены в эфире |

## Research: смена разрешения/соотношения RTMP на лету (2026-06-28)

Вывод: **менять разрешение/соотношение на активном RTMP-соединении — рискованно и не
поддерживается как штатная практика.**
- YouTube определяет параметры энкодера (разрешение, fps) **на старте** трансляции; рекомендован
  постоянный CBR и фиксированное разрешение на весь эфир.
- «Changing the source encoding configuration while using a same opened channel can cause issues» —
  т.е. смена конфигурации энкодера на том же открытом канале может оборвать/испортить поток.
- Для смены формата корректный путь — остановить и пересоздать трансляцию.

→ Полностью подтверждает решение «во время эфира поворот блокируем».

## План реализации

1. **`RtmpStreamer`** — хранимый угол `videoRotation` (StateFlow), `setVideoRotation(deg)`:
   - swap размеров энкодера для превью (`setEncoderSize`) + рестарт превью для реинициализации GL;
   - `prepareVideo` при старте стрима получает обменянные размеры при 90/270;
   - `applyVideoRotation()` (= `setCameraOrientation(deg)`) переживает рестарты превью/стрима/standby;
   - **блок смены угла, пока `isStreaming` == true** (no-op + лог/snackbar).
2. **`StreamingRepository` / `StreamViewModel`** — проброс `videoRotation` + `setVideoRotation`.
3. **UI** — раскрывающееся радиальное меню углов на главном экране; **неактивно во время эфира**.
4. Тест на устройстве: превью 16:9 → выбор 90° → превью 9:16 портрет; Go Live → поток портретный.

## Открытые вопросы / на будущее

- Локальная запись в файл (MediaMuxer / RootEncoder record API) в выбранной ориентации — отдельная фича.
- Если когда-нибудь понадобится смена формата в эфире — только через рестарт RTMP-сессии (см. research).
