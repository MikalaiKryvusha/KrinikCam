# Idea 17 — Запись видео + фото в файл (будущая фича)

> Статус: 📅 ПЛАН (не начато). Заметки/решения собираем здесь, чтобы не потерять контекст.
> Связано: Idea 10 (запись энкодера в файл — уже есть как дев-харнесс), interview_004 (заглушка на разрыв),
> Idea 11 (публикация записи в DCIM/KrinikCam через MediaStore — пайплайн вывода уже готов).

## Суть

- **Запись видео в файл** на устройство (одновременно со стримингом или отдельно).
- **Фото-снимок** (захват одного кадра в файл).
- Вывод — в публичную галерею (`DCIM/KrinikCam`) через MediaStore (пайплайн из Idea 11 готов).

## ✅ Проверено: 4К-запись РАБОТАЕТ (2026-06-29)

Чистая запись 4К (3840×2160) виртуальной камеры в файл — **валидна и читается**: 136 кадров за 6.7с
(~20fps), резкий кадр, корректный MP4. **4К сам по себе НЕ тяжёлый и не падает** (аппаратный энкодер
MediaCodec тянет 4К; стрим 4К на YouTube тоже работает). Фото = захват одного кадра — тривиально.

## ⚠️ КЛЮЧЕВОЙ НЮАНС (помнить при реализации)

При записи в файл **подмена видео-источника на лету ломает `MediaMuxer`** — таймлайн/индекс в MP4
портятся (наблюдали: запись 4К + подмена live→freeze→standby → битый файл, ~5 кадров вместо сотен).
Это НЕ про 4К — это про `changeVideoSource()` во время активной записи в файл.

### Решение (Криник, 2026-06-29)

- **Заглушка / заморозка (interview_004) имеет смысл ТОЛЬКО для СТРИМА** — там нужно держать RTMP
  живым при разрыве камеры (нет файлового muxer'а — подмена источника безопасна).
- **Для записи в файл — НЕ подменять источник и НЕ показывать заглушку.** При отключении камеры во
  время записи — **корректно финализировать (закрыть) уже записанный файл** (валидное видео до момента
  разрыва остаётся в памяти устройства). Никакого битого файла, никакой «Please stand by» в записи.
- Т.е. поведение на разрыв камеры РАЗНОЕ: стрим → freeze→standby (поток жив); запись → чистый stop +
  финализация файла.

### Технически (на будущее)

- Если запись и стрим идут ОДНОВРЕМЕННО и камера отвалилась: стрим уходит в freeze/standby, а запись
  параллельно корректно закрывается (или продолжается со standby-кадром — решить тогда; по Кринику
  предпочтительно закрыть чистый файл). НЕ дать `enterStandby`/`changeVideoSource` испортить активный
  `MediaMuxer`.
- Возможно: при старте записи запоминать, и в обработчике разрыва ветвить — `isRecording` → finalize;
  `isStreaming` → standby.

## 🟡 Реализовано финализирование при разрыве (dayloop 2026-06-30) — проверка блокирована bug 20

Сделано: в `RtmpStreamer.enterStandby()` добавлена ветка — если идёт ЗАПИСЬ (а не стрим), на отрыв
камеры НЕ подменяем источник (это бьёт MediaMuxer), а чисто финализируем файл (`stopRecordToFile()`).
Запись и стрим взаимоисключающи (`startRecordToFile` игнорит старт при активном другом) → ветка
покрывает кейс записи целиком. Заглушка/freeze остаётся ТОЛЬКО для стрима.

⚠️ **Сквозная проверка на харнесе НЕ завершена:** при отрыве источника (`SET_VIRTUAL_CAM off`)
приложение падает нативным SIGABRT (HWUI `EGL_BAD_ALLOC` / «context with no surface») — это
ПРЕД-СУЩЕСТВУЮЩИЙ краш превью при смене источника, см. **`bugs/20_*`**. Он же мешает увидеть чистую
финализацию (процесс умирает). Фикс bug 20 разблокирует проверку idea 17. Код финализации логически
верен (ранний return + stopRecordToFile), но помечаю как непроверенный до закрытия bug 20.
