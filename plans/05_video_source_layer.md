# План 05 — Слой «источник видео» с выбором источника (Phase 4, idea 20)

> **Статус:** 🔧 В РАБОТЕ (2026-07-06). Источник: явное видение Криника (чат 2026-07-06, зафиксировано
> в `ideas/20_add_source_chooser.md`): «Никаких выбор по приоритету. Есть слои. Тип слоя — источник
> видео. Добавляешь в сцену. В настройках слоя выбираешь источник: селфи / фронталка / UVC (если видна)».
> Устройство: Titan 1 @ 192.168.50.187.
> **Требование Криника:** в плане ОБЯЗАТЕЛЬНЫ пункты тестирования инструментами автоматизации.

---

## 0. Архитектурное решение (2026-07-06, чат Криника) — ПРИНЯТО

**Развилка:** камеры (виртуалка / UVC / встроенные ОС-камеры) — это ОДИН тип слоя «Устройство
захвата видео» с выбором конкретного устройства в свойствах? ИЛИ три отдельных типа слоя
(«Виртуальная камера» / «UVC-камеры» / «Встроенные камеры»)?

**РЕШЕНИЕ: ОДИН тип слоя «Устройство захвата видео» (Video Capture Device).** Конкретное устройство
(виртуалка / любая UVC / любая встроенная камера ОС) — полиморфное свойство `source` этого слоя.
Критерий выбора — не «как сейчас», а что проще поддерживать/расширять и что менее багованно.

**Обоснование (оси Криника: А поддержка/расширение, Б меньше багов):**
1. **Тип слоя = поведение РЕНДЕРА, источник = способ ДОБЫЧИ кадров.** Для композитора все три камеры
   идентичны: OES-текстура → квад с `LayerTransform`. Отличается лишь opener (Uvc/Device/Virtual) —
   и он УЖЕ вынесен в стратегию `CameraOpener` (`CameraLayerOpeners.kt`). Три типа слоя = дублировать
   всю слоевую машинерию (трансформа, видимость, z-order, UI-панель, будущая сериализация профиля)
   поверх одинаково рисующихся сущностей → тяжелее поддержка (А✗), больше багов (Б✗).
2. **Эффект второго порядка (расширение).** Новый backend (2-я камера, NDI, RTSP, захват экрана) с
   одним типом = добавить вариант `CaptureSource` + маппинг в opener; слоевая логика (мультиинстанс,
   жесты plans/03, сериализация) не меняется. С тремя типами — переписывать панель/жесты/сериализацию.
3. **Меньше багов при смене источника (Б).** Переключение устройства ВНУТРИ слоя сохраняет
   id/позицию/размер/порядок. Три типа → смена «встроенная→UVC» = удалить+добавить+перепозиционировать
   (потеря трансформы, churn состояния). Так же в OBS: один «Video Capture Device».

**Взято из идеи трёх типов (реальное улучшение):** встроенная камера — НЕ тумблер front/rear, а
ПОЛНЫЙ список родных камер ОС (селфи / основная / ширик / сверхширик / макро / микроскоп — всё, что
отдаёт `CameraManager`). Выбор конкретной — в свойствах слоя.

**Модель:**
```
Layer.VideoCapture(source: CaptureSource, transform, visible, name, id)   // ЕДИНСТВЕННЫЙ тип камеры

sealed CaptureSource:               // лёгкий доменный тип в :feature:streaming (модуль без backend)
  Builtin(cameraId, displayName)    // выбор из ПОЛНОГО списка родных камер ОС
  Uvc(deviceId, displayName)        // выбор конкретной подключённой UVC-вебки
  Virtual                           // дебаг тест-паттерн
  None                              // чёрная база
```
Маппинг `CaptureSource → CameraOpener` живёт в `:app` (зависит от usb+capture+streaming; streaming от
capture НЕ зависит — граница модулей сохранена). Будущие «Браузер»/«Видеофайл»/«Текст»/«Изображение» —
ОТДЕЛЬНЫЕ типы слоя (рисуются иначе); все КАМЕРЫ — один тип (рисуются одинаково).

## 1. Целевая модель
- Камера-слой (`Layer.VideoCapture`) имеет ЯВНО ВЫБРАННЫЙ источник `CaptureSource` (см. §0):
  `Builtin(cameraId)` / `Uvc(deviceId)` / `Virtual` / `None`.
- НЕТ авто-выбора по приоритету (уже убран: DeviceManager не лезет на встроенную; промежуточно
  авто-берёт UVC). Финально — источник задаёт ПОЛЬЗОВАТЕЛЬ в настройках слоя.
- Реестр доступных источников (`DeviceManager.uvcSources` + `phoneCameras` — ПОЛНЫЙ список ОС-камер) —
  показываем как список выбора. UVC — только если реально подключена и видна приложением.

## 2. Scope
- **Фаза A (этот план):** у камера-слоя — выбор источника (Front/Rear/UVC/None) в панели «Слои».
  Один камера-слой. Заменяет авто-приоритет явным выбором.
- **Фаза B (позже, idea 20):** НЕСКОЛЬКО экземпляров одного типа «Устройство захвата видео»
  одновременно (Криник 2026-07-06: слой №1 источник=UVC, слой №2 источник=фронталка — оба в кадре,
  напр. PiP). Модель `Scene(List<Layer>)` это уже допускает; нужно ТОЛЬКО в композиторе: своя
  OES-текстура + свой `CameraOpener` на КАЖДЫЙ экземпляр (сейчас один камера-SurfaceTexture). Плюс
  кнопка «Добавить источник» с типами (устройство захвата / картинка / видео / текст).
  В общем случае число экземпляров НЕ ограничено; на практике — мягкий потолок ~10 (больше не осилит
  процессор). Ограничение можно ввести на уровне UI/сцены при добавлении слоя.

## 3. Шаги реализации (Фаза A)

- [ ] **S1. Модель источника слоя.** Ввести `sealed CaptureSource` (§0: `Builtin(cameraId,name)` /
      `Uvc(deviceId,name)` / `Virtual` / `None`) в `:feature:streaming/scene`. Переименовать
      `Layer.Camera` → `Layer.VideoCapture` (name по умолчанию «Устройство захвата видео») с полем
      `source: CaptureSource`. Обновить `Scene` (when-ветки, `default()`), `withVisible/withTransform`,
      добавить `withSource`. Пробросить в `RtmpStreamer`/UI.
- [ ] **S2. Реестр доступных источников для UI.** `DeviceManager` уже отдаёт `uvcSources`/`phoneCameras`.
      РАСШИРИТЬ enumeration встроенных камер в `:app` до ПОЛНОГО списка `CameraManager` (не только
      front/rear — все ОС-камеры). Добавить `availableSources: StateFlow<List<VideoSource>>` (все
      встроенные + подключённые UVC + Virtual) для меню выбора.
- [ ] **S3. Применение выбора.** При смене `source` слоя: маппинг `CaptureSource → CameraOpener` в
      `:app` → `DeviceManager.selectVideoSource(...)` → `MainScreen` LaunchedEffect открывает нужный
      opener (Uvc/Device/Virtual). Смена корректно закрывает предыдущий opener и открывает новый (без
      churn/краша, риск bug 28 — см. §6).
- [ ] **S4. UI выбора в панели «Слои».** В `StreamLayersOverlay` у слоя видеозахвата — строка
      «Источник: <текущий>» с тапом → список доступных (каждая встроенная камера по имени / каждая UVC
      по имени / Virtual / None). Выбор → S3.
- [ ] **S5. Харнес-команда** `select-source builtin <cameraId>|uvc|virtual|none` (CMD-receiver +
      `ui.mjs cmd`) — тест выбора источника автоматизацией (не только пальцами).
- [ ] **S6. Слои → FAB.** Вынести «Слои» из меню в отдельный МАЛЕНЬКИЙ FAB внизу-слева экрана
      (Криник 2026-07-06). Тап по FAB открывает панель слоёв.
- [ ] **S7 (Фаза A-хвост, заглушка). Фейд-в-заглушку как СОСТОЯНИЕ слоя.** Когда `source` временно
      перестаёт кормить кадры (отвал UVC / провал негоциации / переоткрытие) — слой видеозахвата сам
      плавно (fade) уходит в заглушку KrinikCam и так же плавно возвращается при возобновлении потока.
      Заглушка = поведение ЭТОГО типа слоя, не отдельный слой/тип (закрывает bug 17, ветку fallback из
      Scene Q4). Детектор «нет кадров N мс» + альфа-кроссфейд в композиторе. Может уехать в отдельный
      под-план, но принадлежит именно этому типу слоя.

## 4. Тестирование инструментами автоматизации (ОБЯЗАТЕЛЬНО)

Для КАЖДОГО источника — прогон через харнес + визуальная/ffprobe-проверка:

- [ ] **T1. Выбор FRONT (селфи).** `node tools/ui.mjs cmd select-source front` → `sleep 3` →
      `node tools/ui.mjs screen` → кадр показывает ФРОНТАЛЬНУЮ камеру (лицо/потолок), не rear/UVC.
      Лог: `Active source updated: Front camera`.
- [ ] **T2. Выбор REAR (фронталка/тыл).** `cmd select-source rear` → screen → кадр с ТЫЛОВОЙ камеры.
      Лог: `Active source updated: Rear camera`.
- [ ] **T3. Выбор UVC (вебка).** `cmd select-source uvc` → screen → кадр с ВЕБКИ (широкий угол/то, на
      что смотрит вебка). Лог: `UVC connected` / `Active source updated: 2K USB Camera`.
- [ ] **T4. Выбор NONE.** `cmd select-source none` → screen → чёрная база + плашка «нет источника»,
      без краша (композитор рисует базу).
- [ ] **T5. Запись каждого источника:** `cmd stream-to-file on; cmd go-live; sleep 5; cmd stop` →
      `adb pull` → `ffprobe` (валидный MP4, 1920×1080) + извлечь кадр (`ffmpeg -ss 3 -frames:v 1`) →
      подтвердить, что записан ВЫБРАННЫЙ источник.
- [ ] **T6. Смена источника без краша:** быстро `front→rear→uvc→front` (циклом) → процесс жив
      (`adb shell pidof`), crash-буфер пуст (`logcat -b crash`). Проверка, что смена openers не роняет
      (риск нативного close-краша AUSBC, bug 28 — минимизировать переоткрытия).
- [ ] **T7. Регресс поворота+FBO:** при активном UVC — `set-rotation 90` → камера непрерывна, аспект
      верный (FBO-рефактор из plans/04 не сломан сменой источника).

## 5. Милстоун (Фаза A)
Пользователь (и я через `select-source`) выбирает источник камера-слоя: селфи / фронталка / UVC /
нет — явно, без приоритета. Каждый источник проверен записью+скрином. Фундамент под мультиисточники
(Фаза B) и жесты слоёв (plans/03).

## 6. Риск
- Смена openers (Uvc↔Device) закрывает камеру → нативный close-краш AUSBC (bug 28). Митигация:
  закрывать аккуратно, не частить; тест T6 это ловит.
- Если Фаза B (несколько OES в композиторе) — отдельный план, не сейчас.
