## Диагноз: 29.2 и 29.3 — ОДИН корень

**Сейчас** SurfaceTexture камеры-слоя создаётся в композиторе с буфером = размеру ХОЛСТА энкодера:
`cameraSurfaceTexture.setDefaultBufferSize(encW, encH)` (`CompositorVideoSource.initGl`). Отсюда две
беды:
- **29.2 (искажение):** когда холст ПОРТРЕТНЫЙ (1080×1920), камера-буфер тоже 1080×1920 → AUSBC тянет
  16:9-кадр камеры в 9:16-буфер → растяг. На старте — транзиентный растяг в момент негоциации
  размеров. Камеру рисуем full-frame квадом, аспект камеры не сохраняем.
- **29.3 (замирание на повороте):** смена размера холста портрет↔пейзаж перезапускает композитор
  (`changeVideoSource`) → его OES-текстура камеры пересоздаётся → `onCameraSurfaceReady(null→new)` →
  камера ЗАКРЫВАЕТСЯ и ПЕРЕОТКРЫВАЕТСЯ (freeze/чёрный, а на флаки-2K вообще застревает).

**Требование Криника = архитектурно верное решение:** камера пишет в СВОЮ SurfaceTexture, размер
которой = НАТИВНЫЙ размер камеры (16:9), НЕЗАВИСИМО от ориентации холста. Композитор её самплит и
компонует (аспект-корректно) в холст любой ориентации. Поворот холста НЕ трогает камеру вообще.

## Фикс 29.2 + 29.3 — отвязать камеру от холста

### Целевая модель
```
Камера-продюсер (AUSBC/Camera2/вирт) → ПОСТОЯННАЯ SurfaceTexture (буфер = нативный 16:9, напр.
  1920×1080, НЕ зависит от поворота) → живёт, пока камера подключена
      ↓ (самплится каждый кадр)
Композитор рисует камеру-слой квадом с АСПЕКТ-КОРРЕКЦИЕЙ (letterbox/fit по её реальному аспекту)
  в холст текущей ориентации (1920×1080 или 1080×1920)
      ↓
Поворот холста = только смена размера ВЫХОДНОЙ EGL-поверхности + матрицы; камера-OES/SurfaceTexture
  НЕ пересоздаётся, камера НЕ переоткрывается
```

### Выбранный подход (2026-07-06): ДВУХПРОХОДНЫЙ рендер через FBO
Чистая архитектура (вместо хрупкого «буфер камеры = размер холста»): **проход 1** — рисуем сцену
(камера в нативном 16:9-буфере + оверлеи, с трансформами слоёв) в фиксированный **16:9 FBO**,
аспект-корректно, БЕЗ поворота холста; **проход 2** — блитим FBO-текстуру в выходной кадр (encW×encH)
с аспект-корректным поворотом холста. Так камера всегда 16:9 (не сжимается), поворот — отдельный
финальный шаг. `resizeCanvasKeepingCamera` тогда меняет только выходной viewport, FBO и камера-буфер
остаются 16:9.

**Прогресс (2026-07-06, офис):** ✅ 29.3 (resizeCanvasKeepingCamera — камера не замирает на повороте).
✅ Задел: `GlQuadRenderer.createFramebuffer/deleteFramebuffer/bindFramebuffer` (FBO-инфраструктура,
компилируется). ⏳ ОСТАЛОСЬ: переписать `CompositorVideoSource.drawFrame` на два прохода (FBO 16:9 →
блит с поворотом), убрать ресайз камера-буфера из resizeCanvasKeepingCamera. ← продолжить со след. сессии.

### Шаги
- [ ] **S1. Камера-буфер фиксирован на нативном размере.** В `CompositorVideoSource` завести
      `cameraNativeW/H` (16:9, дефолт 1920×1080; в идеале — реальный негоциированный размер камеры,
      если доступен). `cameraSurfaceTexture.setDefaultBufferSize(cameraNativeW, cameraNativeH)` —
      БОЛЬШЕ НЕ завязывать на encW/encH.
- [ ] **S2. Аспект-корректная отрисовка камеры-слоя.** В `layerMatrixOf`/draw камеры домножать
      позиционную матрицу на масштаб «fit камеры (16:9) в холст с учётом аспекта холста» — чтобы в
      портретном холсте камера не растягивалась, а вписывалась (letterbox или fill-crop — по
      умолчанию, вероятно, «камера на весь портретный кадр = кроп по бокам» ИЛИ «вписать с полями» —
      УТОЧНИТЬ у Криника, но дефолт OBS = crop-to-fill; можно как трансформа слоя).
- [ ] **S3. Live-resize холста БЕЗ переоткрытия камеры (ядро 29.3).** Новый метод
      `CompositorVideoSource.resizeCanvas(newW, newH)` на GL-потоке: пересоздать ТОЛЬКО выходную
      EGL window surface под новый размер (из новой encoder-Surface), обновить `encW/encH`, НЕ трогать
      камеру-OES, `cameraSurfaceTexture`, `renderer`, залитые картинки. НЕ звать `onCameraSurfaceReady`.
      → камера продолжает писать в ту же SurfaceTexture, поток не рвётся.
- [ ] **S4. Провод в RtmpStreamer.** `resizeCanvasInPreview` (уже есть, bug 27) заменить: вместо
      `changeVideoSource(compositor)` (рестарт) вызвать `compositorSource.resizeCanvas(encW, encH)` +
      `setCanvasRotation`. Ключевой вопрос к RootEncoder: как отдать композитору НОВУЮ encoder-Surface
      без `changeVideoSource`/`start`. Варианты: (a) `setEncoderSize` внутри пересоздаёт Surface —
      подписаться на неё; (b) если API не даёт — держать выходную Surface стабильной и делать resize
      средствами GL. ИССЛЕДОВАТЬ RootEncoder GlInterface на старте реализации.
- [ ] **S5. Убрать транзиентный растяг на старте (29.2).** Пока камера не отдала стабильный первый
      кадр — не показывать её слой (рисовать чёрную базу), либо буфер камеры сразу 16:9 (S1) →
      растяга в принципе нет. Проверить на реальном старте.
- [ ] **S6. Верификация:** запись+ffprobe на всех 4 углах — камера НЕ мигает/не застревает при
      повороте (poток непрерывный), аспект камеры корректный (круг круглый) и в портрете, и в пейзаже.
      Живьё: Криник крутит розовую кнопку — видео не замирает.

**Риск:** S3/S4 — GL/RootEncoder surface lifecycle (тот же класс, что bug 27). Делать аккуратно,
проверять каждый шаг записью. Если RootEncoder не даёт resize без рестарта источника — план Б:
держать выходной холст энкодера в превью фиксированным-максимальным и композитить в него (аспект
рулит матрица), а истинный портретный размер ставить лишь на go-live (там пере-attach превью
однократно, не на каждый поворот).

## Фикс 29.1 — множественные диалоги USB-разрешения

**Root cause:** `UsbViewModel` на `DeviceAttached` → `repository.requestPermission(device)` →
AUSBC/USBMonitor всегда показывает системный диалог. `DeviceAttached` может прилетать много раз
(AUSBC/hot-plug churn) → стопка диалогов. Манифест-автогрант (bug 09) покрывает только запуск через
`USB_DEVICE_ATTACHED`, но НЕ горячее переподключение при уже открытом приложении.

### Шаги
- [ ] **S1. Проверять `hasPermission` ДО запроса.** В `UsbDeviceRepositoryImpl.requestPermission`
      (или в `UsbViewModel` перед вызовом): если `usbManager.hasPermission(device)` — сразу открывать
      камеру, НЕ звать `requestPermission` (нет диалога). Грант персистентен на устройство (VID/PID).
- [x] **S2. Дебаунс повторных запросов (2026-07-06).** `UsbDeviceRepositoryImpl.requestPermission`
      гасит дубликаты для того же `deviceId` в окне 3с → стопка одинаковых диалогов не растёт.
- [x] **S3. Манифест-автогрант ПРОВЕРЕН (2026-07-06).** 2K-камера = `class 239 / subclass 2` (Misc/IAD),
      а `device_filter.xml` содержит `<usb-device class="239" subclass="2"/>` → камера покрыта,
      Android помнит грант при запуске через `USB_DEVICE_ATTACHED`. Менять не нужно. Остаток 29.1 =
      hot-plug при УЖЕ открытом приложении (автогрант применяется только на запуск) — S1 (hasPermission).
- [ ] **S4. Верификация:** воткнуть/выткнуть камеру несколько раз — диалог максимум ОДИН раз на
      камеру (в идеале ноль после первого гранта). Криник проверяет hot-replug.
