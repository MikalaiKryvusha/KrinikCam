Для того, чтобы исправить баг, нужно:

- Сфокусироваться только на этом баге
- **Вспомнить опыт ДО начала**: `grep` по `EXPERIENCE.md` тегами бага (например `#usb`, `#rotation`) —
  не повторять подход, о провале которого уже есть запись (навык `/experience`).
- В процессе работы хоть немного но пиши естественным языком в чат о том, что ты сейчас делаешь, чтобы я хоть иногда видел и понимал, в каком процессе мы находимся.
- По всем, даже мелким багам в процессе работы над ними, рефлексируй и собирай знания в отдельных mdдокументах в директории bugs.
- **Гейт намерения перед ПЕРВОЙ правкой поведения** (fable-method, шаг 4; KAIF 1.5): одной строкой —
  `INTENT: код делает <X>; падающий тест/задача ожидает <Y>; спека (README/доки/docstring) говорит <Z>`
  — третий слот заполняй, реально ОТКРЫВ спеку. Если X/Y/Z расходятся — само расхождение и ЕСТЬ
  находка: «баг» может жить в тесте или в постановке задачи, а не в коде. Никогда молча не подгоняй
  одну сторону под другую; порядок авторитетов: явное слово Криника > спека > тесты > текущее
  поведение кода.
- Зайти в цикл: Запуск приложения - воспроизвести баг - почитать логи по багу - попытаться понять его причину - внести точечные единичные изменения в код в попытке исправить баг - собрать билд - опять запустить приложения - опять попытаться воспроизвести баг.
- Суть: Вносятся точечные изменения, затем запускается билд для тестирования того, помогли эти изменения или нет.
- Упрощённо: Фиксинг - Тестирование - Чтение логов - Фиксинг - тестирование - Чтение логов - Фиксинг - Тестирование - Чтение логов - Фиксикнг - и так, пока не заработает коррекно - что и будет критерием приёмки, считаем, что баг исправлен. «Работает корректно» — это **НАБЛЮДЕНИЕ** (запустилось, отрендерилось, посчиталось), а не вывод из чтения диффа: неподтверждённое «исправлено» — классический фрод, который ловит `/fable-judge`.
- **Чек близнецов после фикса** (fable-method, шаг 5c; KAIF 1.5): дефект, найденный в одном месте,
  считается ПОВТОРЁННЫМ где-то ещё, пока не поискал. Назови точную ошибочную конструкцию, поищи её по
  всему проекту и зафиксируй в документе бага (и в отчёте):
  `TWINS: искал <паттерн> — найдено <N> других мест: <файлы, или «нет»>`.
  Почини их или перечисли — заявление о полноте без поиска за ним пусто.
- Для того, чтобы пофиксить баг, часто полезно поискать в интернете его решение на форуме, в Github, reddit

> ⚠️ **ПРАВИЛО 3 ПОПЫТОК → переходи в исследование (`/bug-research`).**
> Если за **три итерации** «точечный фикс → билд → тест» баг НЕ исправлен — ПРЕКРАТИ идти вслепую и
> тыкать наугад. Дальнейшие случайные попытки тратят время, билды и (если тест требует внешних
> сервисов вроде YouTube) ненадёжны. Вместо этого запусти навык **`/bug-research`**: глубокий гуглёж
> с записью сырой базы знаний в документ бага, чтение и анализ кода БЕЗ правок для поиска причины,
> рефлексия и формулировка обоснованной гипотезы + плана. Возвращайся к фиксу только когда понимаешь
> причину. Навык: `.claude/skills/bug-research/SKILL.md`.
>
> 🧠 **И ГЛАВНОЕ (см. `PHILOSOPHY.md`):** долгий затык почти всегда значит, что ты делаешь ФИКС
> СЛИШКОМ СЛОЖНО, потому что НЕ ПОНЯЛ задачу — а НЕ что задача сложная или библиотека «глючит».
> Библиотеки просты, всё придумано до нас. Если фикс громоздкий, с кучей флагов и обходов — это
> красный флаг: остановись и ПЕРЕОСМЫСЛИ задачу простыми словами, найди штатный простой путь
> (KISS + Бритва Оккама). Затык = сигнал упростить понимание, а не «копать сложнее».
- Для эффективного фиксинга, код нужно КОММЕНТИРОВАТЬ.
- Для эффективного фиксинга нужно ОБЯЗАТЕЛЬНО рефлексировать и записывать свои знания по работе над фиксинго бага в отдельный документ именно по этой работе по фиксингу этого бага. Такие руминации можно и нужно заводить отдельными md документами в специальной директории /Users/kryvusha/ai_sandbox/KrinikCam/bugs

- **После закрытия бага (или значимого провала попытки)** — помимо документа в `bugs/` зафиксируй

## UI Automation Tool — инструмент для тестирования

**Инструмент создан: `tools/ui.mjs`**
**Полное руководство: `tools/UI_AUTOMATION_GUIDE.md`**

Никогда не угадывай координаты из скриншотов. Всегда используй:

```bash
# Что на экране?
node tools/ui.mjs dump

# Тапнуть кнопку по тексту (точные координаты из DOM)
node tools/ui.mjs tap "go live"
node tools/ui.mjs tap "menu"

# Проверить наличие элемента
node tools/ui.mjs find "platforms"
```

**Как это работает:**
1. `adb shell uiautomator dump` → XML с `bounds="[x1,y1][x2,y2]"` для каждого элемента
2. Парсится `text`, `content-desc`, `resource-id` — матч по частичному совпадению
3. Вычисляется центр `((x1+x2)/2, (y1+y2)/2)` → `adb shell input tap`

**Типичный тест-цикл баг-фиксинга:**
```bash
# 1. Запустить приложение
adb -s 192.168.1.3:5555 shell am start -n com.kriniks.kcam.debug/com.kriniks.kcam.MainActivity
sleep 3

# 2. Открыть меню и нажать Go Live
node tools/ui.mjs tap "menu" && sleep 1.5 && node tools/ui.mjs tap "go live"
sleep 2

# 3. Сразу читать логи
adb -s 192.168.1.3:5555 pull /sdcard/Android/data/com.kriniks.kcam.debug/files/logs/kcam_$(date +%Y-%m-%d).log /tmp/kcam_today.log
grep -E "startStream|RTMP|Broken|blocked|crash|error" /tmp/kcam_today.log | tail -40

# 4. Смотреть крэш-файлы если приложение упало
adb -s 192.168.1.3:5555 shell ls /sdcard/Android/data/com.kriniks.kcam.debug/files/crash_*.txt 2>/dev/null
```

**ВАЖНО при работе с логами:**
- Всегда проверяй текущее время (`date`) и время последней записи в лог-файле
- Не читай вчерашние логи — используй свежую дату
- Формат пути: `/sdcard/Android/data/com.kriniks.kcam.debug/files/logs/kcam_YYYY-MM-DD.log`
